## 0. Цель MVP

Запустить рабочий калькулятор, который:

1. Проводит клиента по короткой анкете (11 шагов).
2. Считает диапазон часов/срока/стоимости **по правилам** (rule-based).
3. Делит результат на **MVP / Full version**.
4. Генерирует AI-summary и список рисков (Claude API).
5. Собирает лид и отдаёт PDF.
6. Показывает менеджеру заявки в админке.

Не цель MVP: динамический конструктор вопросов, личный кабинет клиента, оплата, CRM-интеграции, мультиязычность, lead scoring, аналитика (см. [§33 ТЗ](plan-ai-calkulyator-stoimosti-razrabotki.md)).

## 1. Scope MVP (in / out)

**В MVP:**
- Лендинг калькулятора + SEO-блок (H1/H2/FAQ).
- Анкета: 9–11 шагов, прогресс-бар, режим «вставить описание проекта».
- Rule-based движок оценки (типы проектов + модули + интеграции + множители).
- AI-слой: классификация проекта, summary, риски, MVP-рекомендация (Claude API, structured JSON).
- Экран результата: summary, breakdown по ролям, MVP vs Full, cost drivers, риски, дисклеймер.
- Lead capture (после 60–80% анкеты).
- PDF-отчёт.
- Email клиенту + email менеджеру.
- Админка: список лидов, карточка лида, настройка ставок и коэффициентов.

**Не в MVP:** drag-and-drop редактор вопросов, lead scoring, аналитика конверсий, мультиязычность, оплата, CRM/ERP-интеграции, личный кабинет.

## 2. Архитектура и стек

Соответствует §29 ТЗ, минимальный набор для MVP:

| Слой | Технология |
| --- | --- |
| Backend | Laravel 12, PHP 8.3 |
| БД | PostgreSQL (или MySQL) |
| Очереди/кеш | Redis (AI-вызовы и email — в очередь) |
| Frontend (калькулятор) | React + Inertia (в составе Laravel-приложения) |
| Стили | Tailwind CSS + дизайн-токены «Revolut-style» (см. §3) |
| Админка | Filament (тема перекрашена под §3) |
| AI | Claude API (`claude-opus-4-8` для анализа ТЗ, `claude-haiku-4-5` для дешёвых классификаций) |
| PDF | Browsershot (Puppeteer) — HTML-шаблон → PDF |
| Email | SMTP / Mailgun, отправка через queue |

> Стек самодостаточен (Laravel + Inertia + Filament) — не нужен отдельный Next.js, что сокращает MVP.

> **Размещение (решено):** калькулятор — **отдельное Laravel-приложение на поддомене** (напр. `calc.<домен>` или `app.<домен>`), со своим деплоем и БД. Текущий статический сайт ([этот репозиторий](../)) не трогаем; на лендинге сайта добавляем ссылку/кнопку на калькулятор. Это чисто разделяет статику и динамику.

## 5. Движок расчёта (rule-based)

Конфиг хранится в БД/JSON (seed), редактируется админом. Формула — §18–19 ТЗ:

```
total = base(project_type)
      + Σ module_hours
      + Σ integration_hours
      + design + qa + devops + pm
total_adjusted = total × complexity × risk
final = total_adjusted + buffer(risk)
```

Выход разбивается по ролям (Backend/Frontend/QA/DevOps/PM/Design/BA) и на MVP vs Full.
Стоимость = часы × ставка из rate card (режимы Economy/Standard/Premium), **в рублях (₽)** — решение §12.2.
Показываем **диапазон**, не точку, + уровень уверенности (±20/35/50%).

Сидируем для MVP: 8 типов проектов, 20–30 модулей, базовые интеграции, множители сложности/риска.

**Сид-данные (решение §12.3):** базовые часы по типам проектов и модулям берём как **плейсхолдеры из ТЗ** (§9, §18, §20), ставки из §21 ТЗ переводим из € в ₽ (см. §21-note). Все значения — в таблицах `estimation_settings` / `estimation_rate_cards`, админ правит без кода. Перед публичным запуском команда заменяет плейсхолдеры на реальные цифры.

**§21-note — ставки в ₽ (стартовые плейсхолдеры):** ставки §21 ТЗ заданы в €; для MVP конвертируем по фиксированному внутреннему курсу (значение курса — параметр в настройках) либо задаём рублёвые ставки напрямую. Режимы Economy/Standard/Premium сохраняем. Точные ₽-ставки подтверждает команда до запуска.

## 6. AI-слой (Claude API)

AI **не считает цену** — только классифицирует и объясняет (§26–28 ТЗ).

- **Вход:** ответы анкеты + (опц.) текстовое описание.
- **Промпт:** §27 ТЗ, structured JSON output.
- **Выход (JSON, §28):** `project_type`, `modules`, `integrations`, `ai_features`, `complexity_level`, `risk_level`, `missing_information`, `mvp_recommendation`, `estimation_notes`.
- Этот JSON → кормит rule-based движок (модули/множители) и текстовые блоки результата (summary, риски, рекомендация).
- Вызовы — в очередь (Redis), с фолбэком: если AI недоступен, показываем чистый rule-based результат без summary.

## 7. База данных (MVP-минимум)

Из §30 ТЗ берём 6 таблиц:

- `estimation_sessions` — сессия расчёта (тип, сложность, риск, диапазоны часов/цены).
- `estimation_answers` — ответы анкеты.
- `estimation_modules` — выбранные модули с часами по ролям.
- `estimation_leads` — контакты + статус (New/Contacted/Qualified/Lost).
- `estimation_settings` — коэффициенты/множители.
- `estimation_rate_cards` — ставки по ролям.

## 13. Definition of Done (MVP)

- [ ] Клиент проходит анкету и видит диапазон часов/срока/стоимости с разбивкой MVP/Full.
- [ ] AI-summary и риски генерируются (с фолбэком при сбое AI).
- [ ] Лид сохраняется, менеджеру уходит письмо, клиенту — PDF.
- [ ] Менеджер видит лид в админке, меняет статус, видит ответы и расчёт.
- [ ] Админ меняет ставки и коэффициенты без кода.
- [ ] Калькулятор и админка выдержаны в Revolut-style дизайн-системе.

## 14. Сценарий тестирования (детальный)

Цель — убедиться, что MVP корректен на трёх уровнях: **движок расчёта** (логика чисел),
**HTTP-флоу** (контроллеры/валидация/редиректы) и **сквозные пользовательские сценарии**
(анкета → результат → лид → PDF/письма → админка). Плюс негативные, граничные, безопасность,
адаптив и регресс.

### 14.0. Подготовка окружения теста
- Отдельная тестовая БД: `DB_CONNECTION=sqlite`, `DB_DATABASE=:memory:` в `phpunit.xml` (миграции на каждый тест).
- Мейлер в тестах — `Mail::fake()`; AI — без `ANTHROPIC_API_KEY` (ветка fallback) и отдельно с `Http::fake()` для ветки Claude.
- Сиды каталога/ставок применяются перед тестами (или фабрики).
- Прогон: `php artisan test`. Ручные сценарии — по чек-листам ниже на `php artisan serve`.

### 14.1. Unit-тесты движка `EstimationEngine` (логика чисел)
| # | Что проверяем | Вход | Ожидаемый результат |
| --- | --- | --- | --- |
| U1 | Минимальный проект | website / detailed_spec / web / без фич / без интеграций / figma_ready | hours_min>0; complexity=simple; risk=low; confidence=high; full≈mvp (мало фич) |
| U2 | Монотонность по фичам | CRM с N фич vs N+1 фич | hours при N+1 ≥ hours при N (добавление функции не уменьшает оценку) |
| U3 | AI повышает риск | одинаковый проект с `ai_features=[]` и с `workflow_automation` | risk_level выше во 2-м; factor больше |
| U4 | Множители из БД переопределяют конфиг | изменить `risk_high` в settings | factor пересчитан по новому значению (после `Setting::flush()`) |
| U5 | MVP ⊂ Full | любой набор с не-MVP фичами (chat, analytics) | mvp.hours_max < full.hours_max; mvp включает только mvp-модули |
| U6 | Полоса уверенности | confidence=high/medium/low | (max-min)/mid ≈ ±20/30/40% соответственно |
| U7 | Стоимость = часы×ставки | задать известные ставки rate_card | price_min/max совпадает с ручным пересчётом по ролям ±округление |
| U8 | Граничные пустые ответы | platforms/features/integrations/ai пустые | без ошибок; считается только база+overhead; risk=low |
| U9 | Неизвестные ключи | features=['нет_такого'] | игнорируются, не падает |
| U10 | Срок адекватен | mid известен | timeline.min≥1, timeline.max≤24, max>min |
| U11 | Разбивка по ролям | любой проект | сумма ролей (mid) ≈ totalMid; backend≥0…pm≥0; design берётся из design_status |

### 14.5. Альтернативные пользовательские сценарии
- **MVP-минимум:** один тип проекта, без фич/интеграций/AI, figma_ready → высокая уверенность, узкий диапазон, MVP≈Full.
- **Максимум сложности:** marketplace + iOS+Android + все фичи + RAG + 3 интеграции + need_design + idea → very_complex/very_high, низкая уверенность, широкий диапазон, срок ближе к 24 мес (с капом).
- **Режим «есть описание»:** заполнено только описание + минимум обязательных шагов.
- **Возврат назад и смена ответов:** изменить тип проекта в середине анкеты → итог пересчитан корректно.

### 14.6. Негативные и граничные
| Кейс | Ожидание |
| --- | --- |
| Прямой POST `/estimate` с `features` = строка, а не массив | валидация `array` отклоняет / приводит, без 500 |
| Очень длинное `project_description` (>5000) | ошибка валидации `max:5000` |
| XSS в полях лида (`<script>`) | Blade экранирует; в админке/письме безопасно |
| SQL-мета символы в ответах | параметризованные запросы, без инъекций |
| `message`/`manager_notes` на границе max | принимается ≤ лимита, отклоняется > |
| Повторная отправка анкеты (double submit) | создаются разные сессии, данные не путаются |
| Доступ к `/result/{чужой-uuid}` | по uuid доступен (публичная ссылка) — это by design; проверить, что нет утечки PII лида до его сохранения |

### 14.8. PDF и письма
- PDF: открывается в Acrobat/Chrome/Preview; кириллица не «крякозябрами» (шрифт DejaVu Sans в dompdf); присутствуют все блоки (оценка, MVP/Full, роли, драйверы, риски, дисклеймер, контакт лида).
- Письмо клиенту: тема, MVP/Full в ₽, ссылка на результат, факторы стоимости.
- Письмо менеджеру: тема с типом/часами, контакты лида, summary, риски, ссылка в админку.
- При сбое отправки (битый SMTP) — флоу не падает (try/catch + `report`), лид всё равно сохранён.

### 14.9. Адаптив и кросс-браузер
- Брейкпоинты: 360 / 768 / 1024 / 1440 px — анкета, результат, админка не ломаются, кнопки кликабельны, таблица лидов скроллится.
- Браузеры: Chrome, Firefox, Safari (вкл. iOS Safari), Edge.
- Alpine: без JS показывается осмысленный фолбэк/деградация (минимум — первый шаг и предупреждение); проверить отсутствие `[x-cloak]`-мерцания.
- Доступность (база): фокус на интерактивных элементах, контраст текста на тёмном фоне, `label` у полей.

### 14.10. Производительность (smoke)
- Время ответа `/estimate` (rule-based, без AI) — десятки мс; с AI — в пределах таймаута 40с, вызов в очереди не блокирует UI (в проде — queue).
- Кэш настроек `Setting::val` не протухает между запросами и сбрасывается при сохранении (`Setting::flush`).
- Лёгкий нагрузочный прогон списка лидов (пагинация по 20) на 1–2k записей.

### 14.11. Целостность данных
- `estimation_answers`: 9 записей на сессию, `answer_label` человекочитаемы.
- `estimation_modules`: по одной на выбранную фичу, часы по ролям совпадают с каталогом.
- `result` (json) консистентен с колонками `total_*` / `mvp_*`.
- Каскадное удаление сессии чистит answers/modules/lead.
