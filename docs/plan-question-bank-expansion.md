> версионирование банка и локализация.

## Часть I. Research: как анкета связана с точностью оценки

### 1. Зачем вопросы вообще нужны движку
Оценка ПО — это превращение **неопределённости** в **диапазон с обоснованием**. Каждый вопрос
существует только если его ответ меняет один из факторов расчёта:

- **+ объём** (модуль/фича → часы по ролям);
- **+ интеграция** (внешняя система → часы + риск);
- **× сложность** (масштаб, число сущностей, ролей, правил);
- **× риск** (неизвестность, legacy, регуляторика, нестабильное API);
- **уверенность** (полнота ответов → ширина диапазона ±20/30/40%);
- **MVP-флаг** (обязательное / второй очереди);
- **ветвление** (открыть/скрыть последующие вопросы).

Правило проектирования: **нет вопроса без эффекта**. Если ответ не двигает ни один фактор —
вопрос не нужен (или это маркетинговая/lead-квалификация, тогда он в отдельном блоке).

### 2. Методологическая база (на чём стоит оценка)
| Подход | Суть | Где используем в калькуляторе |
| --- | --- | --- |
| **Bottom-up / декомпозиция** | проект = Σ модулей/задач | основной метод: модули × роли (§18 ТЗ) |
| **Аналогия** | оценка по похожим прошлым проектам | базовые диапазоны по типу проекта (`project_types`) |
| **Parametric (COCOMO-подобно)** | часы = f(размер) × коэффициенты среды | множители complexity/risk/buffer |
| **Function Points / сущности** | объём ∝ числу сущностей, экранов, интеграций | type-specific вопросы про кол-во сущностей/ролей |
| **Three-point / PERT** | (min + 4·mode + max)/6, диапазон | мы храним min/max, диапазон от уверенности |
| **Story points / T-shirt** | относительная сложность | уровни сложности модулей (S/M/L) |

Вывод для банка вопросов: чтобы оценка была параметрически устойчивой, нужно собирать
**счётные драйверы** (сколько сущностей, ролей, типов документов, интеграций, языков, складов…),
а не только «да/нет». Поэтому в банк добавляем **числовые** и **шкальные** вопросы, а не только списки.

### 3. Принципы адаптивной (ветвящейся) анкеты
- **Type-first**: сначала тип системы → затем подгружаем его трек вопросов.
- **Progressive disclosure**: показываем углублённые вопросы только если включён родительский блок
  (выбрал «нужны платежи» → открываются вопросы про эквайринг/возвраты/рекуррент).
- **Core + Vertical + Conditional**: общее ядро (для всех) + вертикальный набор (по типу) + условные подвопросы.
- **«Не знаю» — валидный ответ**: не блокирует, но снижает уверенность и повышает риск.
- **Бюджет внимания**: видимых обязательных вопросов на сессию — 12–18; остальное — условно/в «расширенном».
- **Умные дефолты**: типовые ответы предзаполнены под выбранный тип (можно изменить).
- **Save & resume**: длинная анкета сохраняется по session uuid.

### 4. Анти-паттерны (чего избегаем)
- Анкета 50+ обязательных вопросов → высокий drop-off (§46 ТЗ).
- Вопросы без влияния на оценку.
- Дубли между core и vertical.
- Жаргон без пояснения (tooltip к каждому неочевидному термину).
- Ложная точность («введите точное число пользователей») вместо диапазонов.

## Часть II. Целевая архитектура анкеты

### 5. Трёхслойная структура трека
```
[CORE]  общие вопросы (тип, стадия, платформы, роли, нагрузка, безопасность, сроки, бюджет, дизайн)
   │
   └── выбран тип = CRM ──► [VERTICAL: CRM]  свои 15–25 вопросов
                              │
                              └── «нужна телефония?» = да ──► [CONDITIONAL] вопросы про телефонию
```
- **CORE** задаётся всегда (≈10–12 вопросов).
- **VERTICAL** — набор под тип системы (12–25 вопросов, многие с умными дефолтами).
- **CONDITIONAL** — раскрываются по ответам (родитель → дети).
- **CROSS-CUTTING блоки** (безопасность, интеграции, нагрузка, данные/миграция) — общие, но их
  состав уточняется типом (например, для ERP по умолчанию включён блок «учёт/документооборот»).

### 6. Типы вопросов (расширяем за пределы single/multi)
| Тип | Назначение | Пример |
| --- | --- | --- |
| `single` | выбор одного | стадия проекта |
| `multi` | набор | какие модули нужны |
| `number` | счётный драйвер | сколько ролей / складов / типов документов |
| `range` | диапазон-бакет | ожидаемая нагрузка пользователей |
| `bool` | да/нет, часто родитель ветки | нужна ли телефония |
| `scale` | 1–5 / низкая-высокая | требования к кастомизации |
| `matrix` | роль × право, модуль × платформа | матрица доступа |
| `ranking` | приоритизация для MVP | что важнее в первую очередь |
| `text` | описание/уточнение (для AI) | опишите специфику бизнес-процесса |

### 7. Условная видимость (rules)
Каждый вопрос/опция может иметь правило показа:
```
show_if: { question: "needs_payments", op: "includes", value: "yes" }
```
Поддерживаем операторы: `equals`, `includes`, `not_includes`, `gt/lt` (для number), `any_of`.
Правила хранятся в БД (редактируемы), вычисляются на фронте (Alpine) и валидируются на бэке.

## Часть III. Модель данных банка вопросов (DB-driven, редактируемый)

> Это развитие §25.5 ТЗ (редактирование вопросов без кода) и переход от `config/questionnaire.php`
> к таблицам. Каталог `config/estimation.php` (часы/множители) остаётся источником коэффициентов,
> но **связь «ответ → эффект»** выносится в БД.

### 8. Таблицы
```
question_groups
  id, key, title, scope (core|vertical|crosscutting),
  project_type (null для core/crosscutting), sort, is_active

questions
  id, group_id, key, type (single|multi|number|range|bool|scale|matrix|ranking|text),
  title, hint, required, default_value, unit (для number),
  show_if (json rule), min, max, step, sort, is_active, version

question_options
  id, question_id, key, label, hint, sort, is_default, show_if (json)

answer_mappings                 -- КАК ответ влияет на оценку
  id, question_id, option_key (null для number/scale),
  effect_type (module|integration|hours|complexity|risk|confidence|mvp_flag|branch),
  effect_target (например module_key='orders' или role='backend'),
  hours_min, hours_max,         -- если effect=hours
  multiplier,                   -- если complexity/risk
  per_unit (bool),              -- для number: эффект × введённое число
  notes

question_set_versions           -- версионирование банка (A/B, откат)
  id, name, status (draft|active|archived), published_at

question_translations           -- локализация (ru/en/...)
  id, translatable_type, translatable_id, locale, field, value
```

### 9. Принцип маппинга «ответ → эффект»
- **single/multi** опция → один или несколько `answer_mappings` (модуль, интеграция, риск…).
- **number** → `per_unit=true`: часы/множитель × число (например, «складов: 5» → ×складской модуль).
- **scale** → масштабирует множитель сложности (1→×1.0 … 5→×1.4).
- **branch** → раскрывает дочернюю группу/вопросы.
- Эффекты **аддитивны**, затем применяются глобальные множители (как сейчас в `EstimationEngine`).

## Часть IV. Общие (cross-cutting) блоки — для всех типов

> Это расширение текущих core-шагов. Многие включаются условно или с умными дефолтами по типу.

### 10. CORE-вопросы (всегда)
1. Тип системы [single] → выбирает VERTICAL-трек.
2. Стадия проекта [single] → риск (idea +25% … detailed_spec 0).
3. Платформы [multi] → web/responsive/iOS/Android/desktop/admin/API/Telegram.
4. Кол-во ролей пользователей [number] → +часы прав/RBAC (per_unit).
5. Тип авторизации [multi] → email/SMS/OAuth/SSO/2FA/magic-link → часы + риск.
6. Ожидаемая нагрузка [range] → ×backend/devops (до 1k … 100k+).
7. Требования к безопасности [multi] → ПДн(152-ФЗ)/платёжные/мед/аудит/шифрование/compliance.
8. Локализация [multi] → один язык / мультиязык / мультивалюта / мультирегион.
9. Данные/миграция [single] → с нуля / импорт из Excel / миграция из старой системы → риск+часы.
10. Дизайн-статус [single] → как сейчас.
11. Сроки [single] и 12. Бюджет [single] — как сейчас (lead-квалификация + подсказка MVP).

### 11. CROSS-CUTTING блоки (общие, уточняются типом)
- **Интеграции** (расширенный список, см. §ниже): платёжные, госсистемы РФ, мессенджеры, аналитика, склад/логистика, бухгалтерия.
- **Нефункциональные**: real-time, очереди, кеш, CDN, горизонтальное масштабирование, SLA/uptime.
- **Отчётность/аналитика**: простые таблицы → BI-дашборды → конструктор отчётов.
- **Уведомления**: email/SMS/push/Telegram/in-app, шаблоны, триггеры.
- **Контент/медиа**: загрузка файлов, видео, документы, версии, антивирус.
- **Поддержка/эксплуатация**: админ-логи, аудит действий, фиче-флаги, мониторинг.
- **Регуляторика РФ** (условно по типу): 152-ФЗ (ПДн), 54-ФЗ (онлайн-кассы/чеки), маркировка «Честный знак», ЕГАИС/«Меркурий», ЕСИА/Госуслуги, ГИС-системы.

### 12.5. Каталог ПРОДУКТОВ калькулятора (что разрабатываем)

> Важно: на сайте перечислены **услуги** («разработка ERP», «внедрение ИИ», «автоматизация
> бизнеса», «аутстаффинг»). В калькуляторе же первый вопрос — это **продукт**: что именно
> получит клиент (конкретная система/приложение). Поэтому каталог типов = **продукты**, а не услуги.

> Vertical-наборы §17a/§18a/§20a/§20b/§23a в этом документе сохранены как **банки вопросов**
> (их можно подключать как отраслевые блоки), даже если соответствующий пункт убран из списка продуктов.

### 13. CRM-система
Драйверы: воронки, источники лидов, автоматизация продаж, телефония, кастомные сущности, права.

1. Кол-во воронок продаж/процессов [number] → ×сложность, +часы (per_unit).
2. Источники лидов [multi] — сайт/звонки/email/мессенджеры/маркетплейсы/реклама/импорт → +интеграции.
3. Управление сделками [multi] — стадии/чек-листы/автоматические задачи/триггеры/прогноз (forecast).
4. Карточка клиента [multi] — история взаимодействий/документы/счета/сегменты/дедупликация.
5. Нужна телефония? [bool] → ветка: какая АТС (Mango/UIS/Asterisk/Sber), запись звонков, click-to-call → +интеграция+часы.
6. Email-трекинг и рассылки [multi] — двусторонняя синхронизация почты/трекинг открытий/массовые рассылки.
7. Автоматизация (workflow) [scale] — нет / простые триггеры / визуальный конструктор бизнес-процессов → ×сложность.
8. Кастомные поля и сущности [single] — фикс-структура / настраиваемые поля / конструктор сущностей → +часы.
9. Права доступа [single] — простые роли / по отделам / иерархия руководитель-подчинённый / по полям → +RBAC.
10. Аналитика продаж [multi] — дашборды/воронка/план-факт/ABC/когорты/конструктор отчётов.
11. Интеграция с 1С/бухгалтерией? [bool] → +интеграция (выставление счетов, синхронизация оплат).
12. Мобильное приложение для менеджеров? [bool] → +платформа.
13. Мультиканальность (омниканальный чат)? [bool] — объединение Telegram/WA/VK/сайт-чат в одно окно → +часы+интеграции.
14. Согласие на обработку ПДн (152-ФЗ) [bool] → +compliance.
15. Объём базы клиентов [range] → ×нагрузка.

### 14. ERP-система
Драйверы: набор модулей, мультиучёт, регламентированный учёт РФ, документооборот, MRP.

1. Какие модули ERP нужны [multi] — финансы/бухгалтерия/закупки/продажи/склад/производство/HR/бюджетирование/казначейство/проектный учёт → каждый = крупный +модуль.
2. Регламентированный учёт РФ (РСБУ)? [bool] → большой +часы + риск (регуляторика).
3. Интеграция/замена 1С? [single] — нет / двусторонняя синхронизация / полная замена 1С → риск×.
4. Мультиюрлицо / мультивалюта / мультисклад [multi] → ×сложность (per_unit где уместно).
5. Документооборот [multi] — типы документов, маршруты согласования, ЭДО (Диадок/СБИС), электронная подпись.
6. Производство [single] — нет / простой учёт / MRP / APS (планирование) → крупный множитель.
7. Складской учёт [multi] — партии/серии/сроки годности/адресное хранение → пересечение с WMS.
8. Закупки [multi] — заявки/тендеры/поставщики/прайс-листы/автозаказ.
9. Бюджетирование и казначейство [bool] → +модуль.
10. Себестоимость и калькуляция [scale] → ×сложность.

5. Хранилище [single] — реляционное / колоночное (ClickHouse) / облачное DWH / data lake → ×сложность.
6. Качество и каталог данных [multi] — валидация/дедуп/Data Governance/линейка (lineage)/каталог.
7. Визуализация [single] — встроенные дашборды / внешний BI (DataLens/Power BI/Superset) / конструктор → +часы.
8. Расчёты и метрики [scale] — простые агрегаты / сложные витрины / семантический слой → ×сложность.
9. ML/предиктив [bool] → ветка на §20 (модели/инференс).
10. Реалтайм-аналитика [bool] → +нефункц. (стрим, окна) → риск.
11. Права на данные (row/column-level security) [bool] → +RBAC.
12. Частота обновления и SLA дашбордов [scale] → ×devops.

### 20a. Digital Health / МИС (медицина)
Драйверы: ЭМК, расписание, телемед, интеграции с гос-МИС, мед-ПДн (особая категория) → высокий риск.

1. Тип решения [single] — МИС клиники / телемедицина / агрегатор записи / носимые/мониторинг / ЛК пациента → база.
2. Электронная мед.карта (ЭМК) [multi] — анамнез/протоколы осмотров/назначения/анализы/документы/шаблоны.
3. Расписание и запись [multi] — врачи/кабинеты/слоты/онлайн-запись/напоминания/листы ожидания.
4. Телемедицина [bool] — видеоконсультации/чат/хранение записей → +часы+риск.
5. Интеграции с госсистемами [multi] — ЕГИСЗ/ЕМИАС/РЭМД/ФРМР/ФРМО/СЭМД → крупный риск× (регуляторика).
6. Рецепты и лекарства [multi] — электронные рецепты/взаимодействия/интеграция с аптеками.
7. Лаборатория (ЛИС) [bool] — заказы анализов/результаты/интеграция с оборудованием → +часы.

9. Мед.ПДн (особая категория, 152-ФЗ) [bool] → обязательный высокий compliance + риск.
10. Доступ медперсонала [single/matrix] — роли врач/медсестра/регистратура/админ → RBAC+аудит (обязательно).
11. Носимые устройства/IoT [bool] → ветка на телеметрию.
12. Сертификация (Росздравнадзор, как МИ) [text] → отдельный риск/сроки → confidence↓.

### 20b. Блокчейн и Web3
Драйверы: сеть, смарт-контракты, кошельки, аудит безопасности, on/off-chain логика.

1. Тип Web3-продукта [single] — токен/NFT / DeFi-протокол / DApp / кошелёк / биржа (DEX/CEX) / частный блокчейн (enterprise) → база.

## Часть VII. Управление длиной и ветвлением

### 25. Как не утопить пользователя
- **Видимых обязательных** на сессию: CORE (≈10) + 6–10 ключевых vertical = **16–20**.
- Остальные vertical/conditional — за тумблером **«Уточнить детали (точнее оценка)»**.
- **Умные дефолты** под тип → пользователь больше подтверждает, чем вводит.
- **Прогресс по секциям** (не по вопросам): «Основное → Специфика CRM → Интеграции → Контакты».
- **«Не знаю»** везде → не блокирует, помечает пробел в `missing_information` (AI это подсветит) и снижает уверенность.
- **AI-ассистент** (опц.): из вставленного описания предзаполняет vertical-ответы → пользователь правит.

### 26. Связь с уверенностью оценки
Чем больше ответов «не знаю»/пропусков в vertical-треке → тем шире полоса (low confidence ±40%)
и тем выше риск-множитель. Полные ответы → high confidence ±20%. Это уже поддержано движком —
банк вопросов просто даёт больше сигналов.

## Часть VIII. Интеграция с движком и текущим кодом

### 27. Что меняется в `EstimationEngine` / каталоге
- Сейчас движок читает `features/integrations/ai_features` как плоские списки.
- Расширение: добавить обработку **vertical-ответов** через `answer_mappings` (БД), где опция →
  эффект (module/integration/hours/multiplier). Движок остаётся аддитивным + множители.
- **number/scale** вопросы → `per_unit`/масштабирование множителя (новая ветка в движке).
- `config/estimation.php` остаётся хранилищем коэффициентов; **что выбрано** и **как влияет** —
  частично переезжает в БД (банк вопросов), чтобы редактировалось без кода (§25.5 ТЗ).

3. Добавить vertical-группы по типам (CRM, ERP, …) и их `answer_mappings`.
4. Включить условную видимость (`show_if`) и number/scale-обработку.
5. Обновить фронт анкеты (Alpine): загрузка трека по типу, условные вопросы, новые типы инпутов.
6. Раздел админки «Конструктор вопросов» (CRUD групп/вопросов/опций/правил/маппингов).

## Часть IX. Authoring, governance, качество

### 29. Кто и как ведёт банк вопросов
- **Владелец** — пресейл/архитектор: отвечает за корректность драйверов и часов.
- **Версионирование** (`question_set_versions`): правки в draft → ревью → publish; откат.
- **Источник истины по часам** — реальные прошлые проекты студии; периодическая калибровка
  (раз в квартал сверять оценку калькулятора с фактическими часами закрытых проектов).
- **A/B анкеты**: сравнивать конверсию/drop-off разных формулировок и длины.

### 30. Метрики качества анкеты
- **Drop-off по вопросу** (где уходят) → кандидаты на упрощение/перенос в «детали».
- **Доля «не знаю»** по вопросу → плохая формулировка или преждевременный вопрос.
- **Точность**: |оценка − факт| / факт по закрытым сделкам → калибровка коэффициентов.
- **Полнота lead-данных** и **корреляция** длины анкеты с качеством лида.

## Резюме
1. Перейти от плоской анкеты к **трёхслойной**: CORE + VERTICAL (по типу) + CONDITIONAL.
2. Вынести банк вопросов и связь «ответ → эффект» в **БД** (редактируемо, версионируемо).
3. Добавить **счётные/шкальные** вопросы (function-point драйверы) — для параметрической устойчивости.
4. Сделать **глубокие type-specific наборы** для всех текущих типов (§13–24) и расширить вертикали (§Часть VI).
5. Управлять длиной через умные дефолты, прогрессивное раскрытие и «не знаю».
6. Калибровать часы по фактическим закрытым проектам; мерить drop-off и точность.
