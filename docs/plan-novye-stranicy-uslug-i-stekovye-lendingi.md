# План добавления новых страниц: услуги, стековые и отраслевые лендинги

Дата: 12.07.2026.
Основание: gap-анализ по [seo-pages/konkurenty-uslugi.md](../seo-pages/konkurenty-uslugi.md) (801 услуга у 33 конкурентов) против текущих 48 страниц услуг в [urls.json](../urls.json): 109 услуг конкурентов у нас отсутствуют полностью, ~170 покрыты лишь упоминанием внутри других страниц.

## Общие правила (обязательные)

- Новая страница = новый каталог `<dir>/index.html` + запись в `urls.json` (и `count` +1). Существующие URL не трогать (CLAUDE.md).
- Страницы собираются генератором `_generate_pages.py` из текстов-источников (`docs/texts-usligi.md`), не руками.
- После каждой волны: регенерация меню «Услуги» и футера на всех страницах, обновление `sitemap.xml`, внутренняя перелинковка (2–3 входящих ссылки с релевантных страниц).
- Контент страницы услуги — по полному циклу скилла `service-page-development` (семантическое ядро → «идеальная страница» по конкурентам → SEO-оптимизация → сборка из библиотеки блоков).

## Трек B. Стековые лендинги — новый раздел `/tehnologii/`

**Зачем:** SEO-тактика конкурентов (novacom — 15+ стековых страниц и 9 лендингов миграций, metalamp — 20+ web3-стековых, у surf Flutter — флагманская страница). Запросы «разработка на X» коммерческие, конкуренция ниже, чем по услугам.

**Решение до старта (URL создаются навсегда):** раздел `/tehnologii/<slug>/` + хаб `/tehnologii/`.
Рекомендация — именно отдельный раздел, а не `/uslugi/`: там уже 48 страниц, и стековые лендинги — другой тип контента. Альтернатива (`/uslugi/<slug>/`) допустима, но слаги согласовать до создания первого URL.

Компания работает со всеми технологиями, поэтому список лендингов не ограничен: приоритет определяется спросом (Вордстат) и частотой у конкурентов. Начинаем с ядра, уже упомянутого в текстах сайта (`docs/texts-usligi.md`): Python (Django/FastAPI), PHP (Laravel), Node.js, React, Vue, Flutter, React Native, Kotlin, Swift, TON/Solidity, PostgreSQL, Kubernetes, — дальше расширяемся на остальные.

### Этап B0 — инфраструктура раздела (разово, до первой страницы)

1. `_generate_pages.py`: поддержка раздела `tehnologii` (сейчас генератор знает только страницы услуг) + новый источник текстов `docs/texts-tehnologii.md`.
2. Единый шаблон стековой страницы из библиотеки блоков внутренних страниц (скилл `inner-page-blocks`): hero → прайс-карточки (Цена/Сроки/Формат) → «что делаем на X» (bento-ряды) → когда X подходит / сравнение с альтернативой (compare-таблица) → кейсы → FAQ → CTA. Отличие от страниц услуг — блок «X ↔ услуги»: ссылки на релевантные страницы `/uslugi/`.
3. Хаб `/tehnologii/` (карта технологий со ссылками) + записи в `urls.json`.
4. Навигация: категория/панель «Технологии» в меню «Услуги» и колонка в футере.

### Волна B1 — основной стек (хаб + 6 страниц)

`/tehnologii/`: `python`, `react`, `flutter`, `nodejs`, `laravel`, `vue`

### Волна B2 — лендинги миграций (перелинковка с `modernizaciya-legacy`)

### Волна B3 — расширение (порядок по Вордстату)

`react-native`, `kotlin`, `swift`, `java`, `golang`, `nextjs`, `nestjs`, `angular`, `django`, `fastapi`, `ton`, `1c-bitrix`, `wordpress`, `unity` — и далее по мере появления спроса: раздел рассчитан на постоянное пополнение.

**Процесс на стековую страницу — облегчённый** (не полный service-page-development): сокращённое семядро (Вордстат + подсказки, скилл `seo-semantic-core`), контент по единому шаблону раздела, сборка генератором, проверка оформления.

## Риски и ограничения

- **URL необратимы** — слаги всех трёх треков согласовать до создания каталогов (особенно имена разделов `/tehnologii/` и `/otrasli/`). Существующие отраслевые страницы (финтех, медицина) остаются в `/uslugi/` навсегда.
- **Каннибализация запросов**: мобильная разработка ↔ iOS ↔ Android ↔ кроссплатформа ↔ Flutter — развести интенты на этапе семядер, прописать перелинковку «общая → узкие».
- **Кейсы**: к моменту публикации страницы желательно иметь 1–2 релевантных кейса в портфолио (для пентеста — учесть требования к ИБ-подрядчикам).
- **Генератор**: `lowPrice` захардкожен — при добавлении страниц с ценами проверить Schema.org-разметку; меню «Услуги» уже плотное — новые пункты добавлять в существующие категории, «Технологии» — отдельной панелью.
