# SEO-семантика Яндекса для конкретной страницы

Собери проверяемое ядро для одного URL и сохрани результат в
`seo-core/<путь-страницы>.md`. Не превращай выдачу Вордстата в готовое ядро: каждый
запрос должен пройти проверку релевантности, интента и целевого URL.

Перед работой прочитай:

- [references/metodika.md](references/metodika.md) — определения, метрики и правила решений;
- [references/shablon.md](references/shablon.md) — обязательный формат результата.

Структурный пример отчёта:
[seo-core/uslugi/razrabotka-erp-sistem.md](../../../seo-core/uslugi/razrabotka-erp-sistem.md).

## Входные данные

1. **URL или путь страницы.** Если не указан и его нельзя однозначно определить — спроси.
2. **Источник страницы.** Сверь локальный `<путь>/index.html` с опубликованным URL, если
   задача относится к production. Зафиксируй, какой вариант анализировался.
3. **География бизнеса.** Не приравнивай «вся РФ» к фильтру «Все регионы». Для этого
   проекта при федеральном продвижении используй «Россия», если пользователь не задал
   другой рынок. Для локального бизнеса нужен целевой регион.
4. **Роль страницы.** Определи предложение, аудиторию, целевое действие и границы услуги.
5. **Доступы.** Проверка в Яндекс Вебмастере обязательна. Если активной сессии нет,
   открой Вебмастер в браузере и попроси пользователя авторизоваться в открытом окне.
   Не проси логин/пароль в чате и не вводи их сам.

## Неподменяемые правила

- Вордстат показывает **число запросов**, а не уникальных людей и не показы страницы.
- Вебмастер показывает отдельные метрики: спрос, показы URL, клики, CTR и позицию.
- Не переходи к сбору масок и Вордстата, пока обязательная проверка Вебмастера не
  выполнена. Если после авторизации нет прав на подтверждённый сайт, остановись.
- Не объединяй числа из разных источников и периодов в одну колонку.
- Не придумывай значения. Если данных нет, пиши `н/д` и причину.
- Базовые числа Вордстата включают вложенные уточнения; не суммируй их.
- Один макрокластер интента назначай одному каноническому URL. У одного URL может быть
  несколько близких подкластеров, соответствующих разделам страницы.
- Не меняй страницу или её URL без отдельного запроса пользователя.

## Рабочий процесс

### 1. Изучи страницу и карту сайта

1. Проверь наличие пути в [urls.json](../../../urls.json).
2. Прочитай Title, Description, H1–H3, первый экран, предложение, цены, этапы, FAQ,
   кейсы, интеграции, отрасли и внутренние ссылки.
3. Прочитай `urls.json` и выдели соседние URL, которым могут принадлежать смежные
   интенты. Не делай вывод о каннибализации только по похожим словам.
4. Запиши 1–3 маркерные формулировки, границы предложения и целевое действие.
5. Если страница недоступна, не индексируется или её содержимое нельзя проверить,
   зафиксируй ограничение; не маскируй его семантическим отчётом.

### 2. Обязательно проверь URL в Яндекс Вебмастере

1. Открой через Playwright/MCP-браузер `https://webmaster.yandex.ru/`.
2. Если появилась форма входа, сообщи пользователю, что окно открыто, и попроси его
   самостоятельно авторизоваться. Приостанови дальнейший сбор и дождись подтверждения
   пользователя, затем проверь, что Вебмастер действительно открылся.
3. Выбери подтверждённый сайт, которому принадлежит целевой URL. Не подменяй его другим
   зеркалом, протоколом или доменом.
4. Если после авторизации сайт отсутствует или у аккаунта нет прав, это блокирующее
   условие: сообщи, что требуется добавить/подтвердить сайт или выдать доступ, и не
   создавай финальное ядро по неполной методике.
5. В **Расширенной аналитике поисковых запросов по URL** получи запрос, показы страницы,
   клики и позицию за выбранный период и регион.
6. В **Мониторинге запросов** получи запросы, по которым URL уже появляется, их CTR,
   позицию и динамику. Для итоговых выводов предпочитай завершённые дневные данные, а
   не предварительные почасовые.
7. В **Подборе поисковых запросов и анализе рынка** выполни подбор по маркеру и по
   группе подтверждённых URL, если интерфейс позволяет; просмотри основные и
   дополнительные запросы, автоматические кластеры, спрос, клики и конкурентность.
8. Зафиксируй сайт, дату, выбранные периоды, регион, устройства и результат каждой
   проверки. Если доступ есть, но у новой/малой страницы нет статистики, запиши
   «проверено, данных нет» и продолжай: обязательной является проверка, а не наличие строк.

Не заменяй этот этап Вордстатом, поисковой выдачей или предположением о том, что данных
нет. При временной ошибке повтори безопасную проверку; если доступ после авторизации всё
равно невозможен, остановись и запроси у пользователя устранение блокировки.

### 3. Составь и расширь маски

Формируй маски из релевантных странице фасетов:

- сущность и синонимы;
- действие/услуга;
- задача пользователя;
- коммерческий сценарий;
- аудитория и отрасль;
- свойства, технология, интеграция;
- миграция или альтернатива;
- география;
- аббревиатуры, профессиональные названия и транслитерация.

Не создавай обязательные комбинации вроде `купить X`, если бизнес-модель им не
соответствует. Не ограничивайся числом масок: продолжай, пока новые релевантные
формулировки появляются, и остановись после смыслового насыщения.

### 4. Собери Вордстат

Выбери наиболее надёжный доступный способ:

1. Wordstat API — для повторяемого или большого сбора при наличии доступа;
2. CSV из интерфейса — предпочтительно для полного ручного экспорта;
3. Playwright/browser snapshot — если выгрузка недоступна.

Для интерфейса открой `https://wordstat.yandex.ru/`, настрой регион и устройства. Для
каждой маски собери «Топы запросов» и «Похожие запросы». Зафиксируй источник, маску,
дату, период, регион и устройство.

Для 3–5 опорных запросов проверь варианты операторов:

- `"фраза"` — фиксирует количество слов, но не порядок;
- `!слово` — фиксирует словоформу;
- `[слова]` — фиксирует порядок;
- комбинируй операторы только когда соответствующее ограничение действительно нужно.

В результате обязательно укажи выражение с оператором рядом с полученным числом. Если
операторная проверка не выполнена, не называй базовое число точной частотностью.

### 5. Нормализуй и классифицируй

Сохраняй исходную формулировку. Для дедупликации создай нормализованное представление,
но не теряй порядок слов и словоформу в исходных данных.

Каждому запросу назначь один статус:

- целевой для страницы;
- поддерживающий для блока страницы;
- передать другому существующему URL;
- кандидат на новую страницу или статью;
- исключить;
- требуется дополнительная проверка.

Не удаляй автоматически информационные, брендовые, опечаточные или нулевые запросы.
Оцени их по роли страницы, бизнес-ценности, сезонности и выдаче; исключённые запросы
сохрани с причиной.

### 6. Проверь интент и совместимость по выдаче

Для спорных и опорных запросов изучи топ-10 органических результатов Яндекса в целевом
регионе и на целевом устройстве. Зафиксируй дату, типы документов и смешанность выдачи.
Не считай рекламные и специальные блоки органическими URL.

При SERP-кластеризации нормализуй URL и укажи метод:

- размер топа;
- hard или soft;
- порог общих URL;
- регион и устройство.

Порог пересечения — эвристика, а не правило Яндекса. Если данные неоднозначны, оставь
запрос в статусе «требуется проверка», а не принимай уверенное решение.

### 7. Построй макрокластер и подкластеры

1. Сформируй макрокластер запросов, совместимых с целевым URL.
2. Раздели его на столько подкластеров, сколько реально нужно странице: основной,
   цена, процесс, отрасли, интеграции, альтернативы и другие релевантные сегменты.
3. Сопоставь каждый подкластер с существующим или рекомендуемым блоком страницы.
4. Сверь запросы с `urls.json` и фактическими URL из Вебмастера. Запрос другой страницы
   вынеси в таблицу передачи, а не оставляй в целевом ядре с одним комментарием.

### 8. Приоритизируй

Не сортируй решения только по базовому числу Вордстата. Оцени:

1. релевантность предложению;
2. совпадение интента;
3. бизнес-ценность;
4. подтверждённый спрос;
5. текущие показы, позицию и CTR URL;
6. конкурентность;
7. контентный пробел;
8. риск каннибализации.

Назначь приоритет `P0`, `P1` или `P2`. Внутри равного приоритета можно сортировать по
сопоставимой метрике спроса.

### 9. Запиши результат

Создай `seo-core/<путь-страницы-без-index.html>.md` по
[references/shablon.md](references/shablon.md). Не смешивай метрики разных источников.
Добавь:

- scope, источники, даты, периоды, регион и устройства;
- результат обязательной проверки Вебмастера и текущую видимость URL; если статистики
  нет после успешной проверки, явно напиши «проверено, данных нет»;
- таблицы целевых подкластеров;
- операторную проверку опорных запросов;
- переданные, исключённые и нерешённые запросы;
- конкретные рекомендации по Title/H1/Description, структуре, FAQ, цене, кейсам и
  перелинковке;
- план мониторинга после внедрения.

Рекомендации должны решать задачи пользователя, а не предлагать списки точных вхождений.
Не задавай фиксированную длину Title/Description как правило Яндекса и не используй
термин «LSI-слова» как фактор ранжирования.

### 10. Проверь качество

- [ ] Число запросов Вордстата не названо показами страницы.
- [ ] Вебмастер открыт под авторизованным пользователем, выбран правильный подтверждённый
  сайт и зафиксирован результат проверки целевого URL.
- [ ] У каждой числовой колонки указан источник, период и тип метрики.
- [ ] Базовые включающие числа не суммируются.
- [ ] Операторные значения сопровождаются использованным выражением.
- [ ] Нет выдуманных значений; отсутствие данных обозначено `н/д`.
- [ ] Интент проверен, а спорные решения помечены уровнем уверенности.
- [ ] Макрокластер не конфликтует с другими URL; подкластеры мапятся на блоки страницы.
- [ ] Исключённые и переданные запросы сохранены с причиной.
- [ ] Рекомендации не ведут к запросному спаму и не обещают позиции.
- [ ] Контент и URL страницы не изменены без запроса пользователя.

## Отчёт пользователю

Укажи путь к файлу, число целевых запросов и подкластеров, запросы `P0`, переданные
интенты, главные рекомендации и ограничения источников. Предложи пересмотр при заметном
изменении предложения/выдачи/спроса либо планово через 3–6 месяцев; для сезонной темы
ориентируйся на полный цикл спроса.
