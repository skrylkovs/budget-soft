# Разработка страницы услуги по полному циклу

Проведи один существующий URL страницы услуги через весь конвейер: семантика →
эталонный контент по конкурентам → SEO-оптимизация контента → структура
страницы с нуля на блоках генератора → готовая страница. Результат —
обновлённая страница `uslugi/<slug>/` с собственной структурой, каждый блок
которой обоснован ядром и анализом конкурентов, плюс артефакты каждого шага в
`seo-core/`, `seo-pages/`, `seo-optimization/`. Структура НЕ срисовывается ни
с одной существующей страницы сайта — она выводится из контент-документа.

Скилл — оркестратор. Каждый шаг выполняется по своему скиллу, читай его перед
шагом:

1. [seo-semantic-core](../seo-semantic-core/SKILL.md) — семантическое ядро;
2. [seo-ideal-page](../seo-ideal-page/SKILL.md) — контент-документ по конкурентам;
3. [seo-page-optimization](../seo-page-optimization/SKILL.md) — методика
   оптимизации, конвейер генератора, верификация и отчёт;
4. [inner-page-blocks](../inner-page-blocks/SKILL.md) — справочник разметки
   блоков внутренних страниц (для проверки, что оформление не потерялось).

Эталон **качества артефактов** (не состава страницы!) — тройки файлов
`seo-core/`, `seo-pages/`, `seo-optimization/` для `razrabotka-crm-sistem` и
`mobilnaya-razrabotka`. Сами страницы этих slug — справка по синтаксису
подключения механизмов генератора, а не шаблон структуры: их состав секций
копировать нельзя.

## Входные данные

1. **Целевая страница.** URL или путь вида `uslugi/<slug>/`. Проверь наличие в
   [urls.json](../../../urls.json). Если страницы там нет — это не разработка
   существующей страницы услуги, остановись и уточни у пользователя.
2. **Библиотека блоков.** Структура собирается с нуля, но из фиксированного
   набора механизмов генератора (таблица «Библиотека блоков генератора» ниже) —
   вёрстка и CSS проекта ограничены этим набором. Новые типы блоков (новая
   вёрстка/CSS) — только по явному запросу пользователя. Донора структуры нет;
   если пользователь явно попросил повторить архитектуру конкретной страницы —
   это отдельное требование, зафиксируй его в отчёте.
3. **Готовые артефакты.** Если `seo-core/uslugi/<slug>.md` и/или
   `seo-pages/uslugi/<slug>.md` уже существуют и свежие (проверь дату анализа и
   совпадение Title/H1 с текущей страницей) — не пересобирай их молча: сообщи,
   что нашёл, и предложи продолжить с существующими либо пересобрать.
4. **Доступы.** Шаг 1 требует авторизации в Яндекс Вебмастере, шаги 1–2 —
   браузера (Playwright MCP) для Вордстата и выдачи. Блокировки доступов
   обрабатывай по правилам соответствующих скиллов, не имитируй данные.

## Неподменяемые правила

- **URL страницы неизменен** (правило CLAUDE.md): не переименовывай директорию,
  не трогай путь `uslugi/<slug>/index.html`. Меняется только содержимое.
- **Структура страницы собирается с нуля.** Состав, порядок и количество
  секций выводи только из контент-документа шага 3 (конкуренты + ядро).
  Существующие страницы услуг сайта (ERP, CRM, мобильная и др.) — справка по
  синтаксису подключения механизмов, а не шаблон: сверять с ними состав секций
  или «начинать с их структуры и дорабатывать» запрещено. Если итоговый состав
  случайно совпал с какой-то страницей сайта — обоснуй каждый блок конкурентами
  и ядром в отчёте, иначе структура не принята.
- **Четыре служебных блока остаются свои: Отрасли, Цифры, Портфолио, Обсудить
  проект.** Генератор выводит их на каждой странице услуги автоматически. В
  терминах разметки: «Отрасли» — `section.ai-directions`
  (`AI_DIRECTIONS_COPY[slug]`), «Цифры» — `section.stats`, «Портфолио» —
  `section.cases`, «Обсудить проект» — `section.cta` (`SERVICE_CTA[slug]`).
  Их словари не переписывай текстами других страниц; улучшать их **свой**
  текст можно только когда этого требует ядро/контент-документ.
- **Правь источники генератора, а не сгенерированный HTML.** Почти весь HTML
  страницы услуги перезаписывается запуском `python _generate_pages.py`; правка
  `index.html` напрямую будет потеряна. Полная таблица «что менять → где
  источник» — в seo-page-optimization, раздел «Конвейер контента этого проекта».
- **Порядок шагов не меняй и шаги не пропускай.** Контент без ядра и анализа
  конкурентов — не результат этого скилла. Исключение — готовые свежие
  артефакты (см. «Входные данные», п. 3).
- **Факты компании — только с сайта или от пользователя.** Цены, сроки, кейсы,
  гарантии не выдумывай; несогласованные цифры несут маркер `[≈ согласовать]`
  из seo-ideal-page — перед публикацией их надо согласовать или убрать.
- **Не обещай позиций.** Итог — обоснованная страница, а не гарантия топа.

## Библиотека блоков генератора

Из этих механизмов собирается структура страницы. Каждому блоку
контент-документа подбери механизм отсюда; блок без спец-механизма — обычная
секция с прозой/бенто-карточками:

| Механизм (тип блока) | Где включается |
|---|---|
| H1 в секции `.revolution` (а не в hero) + Service/Offer/FAQPage JSON-LD | `SEO_ENHANCED_SLUGS[slug]` в [_generate_pages.py](../../../_generate_pages.py) (`service_type`, `low_price`) |
| Секция «О решении» с картой `erp-map` (заголовок + лид + подзаголовок «когда оправдано» + чек-лист сигналов + иллюстрация) | `SECTION_BODY_RENDERERS[(slug, "О решении")] = make_about_body_renderer(...)` + секция `{section: О решении}` в MD |
| Карточки «Цена / Сроки / Формат оплаты» внутри `erp-map__signals` | `SERVICE_PRICING[slug]` + slug в `PRICING_IN_SIGNALS` |
| Пакеты стоимости карточками `.page-price-cards` | маркер `{price-cards}` перед таблицей в MD |
| Сравнительная таблица `.compare--service` | маркер `{compare}` перед таблицей в MD |
| Ряды бенто-карточек по два в ширину | `SERVICE_BAND_ROWS[slug]` — группы заголовков `###` из MD |
| Иконки карточек «Почему мы» (`images/why/*.png`) | `SERVICE_BAND_ICONS` — по точному тексту заголовка карточки |
| Блок «Частые вопросы» + FAQPage-разметка 1:1 | `SERVICE_FAQ[slug]` |
| Title / Description / Keywords | `SEO_TITLES[slug]`, `SEO_DESCRIPTIONS[slug]`, `SEO_KEYWORDS[slug]` |
| Структура и текст секций | секция услуги в [docs/texts-usligi.md](../../../docs/texts-usligi.md): `**Title:**`, `**H1:**`, лид, затем блоки `{section: <Бровь>}` + `### Заголовок`; `{icon: имя}` в пунктах списков модулей |

Состав и порядок секций свободны — их диктует контент-документ шага 3.
Записи реализованных slug (`razrabotka-erp-sistem`, `razrabotka-crm-sistem`,
`mobilnaya-razrabotka`) в словарях используй **только** как образец синтаксиса
подключения механизма (греп по slug), не как шаблон состава страницы.

### Точные контракты генератора (чтобы не перечитывать код)

Проверено на прогонах CRM (04.07.2026) и мобильной разработки (05.07.2026):

- Секции `docs/texts-usligi.md` сопоставляются со slug **по порядковому номеру**
  `## N.` (массив `SERVICE_META` в `_uslugi_data.py`), а не по названию:
  заголовок секции менять можно, номер и порядок — нет.
- Лид страницы = первый абзац секции **до первого заголовка**; у enhanced-slug
  он выводится под H1 в `.revolution`, hero-заголовок понижается автоматически.
- «О решении» (`make_about_body_renderer`): после `###`-заголовка **ровно два
  абзаца** (первый → лид шапки, второй → intro, он обязан оканчиваться текстом
  `<signals_heading>:`), затем список. Любое отклонение — рендерер молча
  оставляет блок прозой без карты и spec-карточек.
- `{compare}`: шапка колонок — `Тег :: Название`; акцент (`--ai`) получает
  **последняя** колонка, остальные — `--legacy`; `data-label` берётся из
  названия колонки автоматически.
- `{price-cards}`: колонки «Пакет | Цена | Состав | Срок»; `Бейдж :: Название`
  в первой колонке делает карточку featured.
- Бенто-ряды `SERVICE_BAND_ROWS` матчатся по **точному тексту** `###`-заголовка;
  карточка вне пар остаётся полноширинной; CSS-сетка ряда — две колонки,
  трёхколоночных рядов нет (нечётное число карточек = пары + полноширинная).
- `{icon: …}` в списках — только из набора: `finance`, `warehouse`,
  `procurement`, `production`, `sales`, `hr`, `bi`.
- Ссылки в MD — корневые (`/путь/`); в HTML попадают как есть.
- `SERVICE_PRICING[slug]` может уже существовать (у мобильной была запись до
  цикла) — проверь перед добавлением; `SEO_TITLES[slug]` не обязателен: Title
  берётся из `**Title:**` секции MD, бренд генератор дописывает сам.
- Штатный шум каждого прогона `python _generate_pages.py`: случайный адрес
  офиса (`_OFFICE_ADDRESSES`) и случайные акцент-цвета карточек workflow меняют
  по 2–8 строк на **всех** страницах — это не контентные правки, построчно не
  разбирать; sitemap при повторной генерации в тот же день не меняется.

## Шпаргалка исполнителя (уроки прогонов 04–05.07.2026)

Конкретика этого проекта, экономящая заходы в браузер и чтение кода. Если
интерфейсы Яндекса изменились — доверяй интерфейсу, а не шпаргалке, и обнови её.

1. **Прямые URL разделов Вебмастера** (хост в пути — `https:budget-soft.ru:443`;
   не гадай адреса, `/query-analytics/` и `/efficiency/analytics/` не существуют):
   - Расширенная аналитика по URL (β): `/site/<host>/efficiency/detailed-analysis/`
   - Мониторинг запросов: `/site/<host>/efficiency/monitoring/`
   - Статистика запросов: `/site/<host>/efficiency/statistics/`
   - Подбор запросов и анализ рынка (β):
     `/site/<host>/efficiency/wordcraft/?device=ALL_DEVICES&mode=QUERY&userQueries=SITES&rivals=GENERAL&query=<маска>`
2. **Выгрузка «Расширенной аналитики по URL»:** кнопка «Подготовить аналитику»
   активируется только после выбора диапазона дат в календаре; вводи **только
   URL подтверждённого хоста** (не-www; чужой хост молча не отправляется);
   «Запросов: N» в подробностях = потраченная квота (дни × ссылки, лимит
   100/день), а не строки данных; отчёт скачивается как `csv.gz` в
   `.playwright-mcp/` — пустой файл после распаковки = «проверено, данных нет».
3. **Блокеры хоста уже задокументированы** (www-зеркало не в Вебмастере, не-www
   отдаёт 301, в индексе только главная — см. seo-core ERP/CRM/мобильной). Для
   нового slug достаточно одного захода на дашборд для сверки актуальности —
   не пересобирай доказательства по всем разделам.
4. **Выдачу Яндекса собирай один раз на шаги 1–2:** те же 3–4 формулировки
   обслуживают и SERP-проверку интента ядра, и список конкурентов для разбора.
   JS-экстракторы выдачи и страниц — в методике seo-ideal-page, работают как
   есть.
5. **Цифры `[≈ согласовать]` — рабочий прецедент CRM/мобильной:** новые пакеты
   цен и длительности вносим в источники сразу, в отчёте шага 6 выделяем блок
   «⚠️ Не подтверждены бизнесом — подтвердить до деплоя» (деплой и так за
   пользователем). Утверждения о ценах конкурентов на страницу не выносим —
   только в приложение А контент-документа.
6. **Верификацию шага 6 делай программно** по сгенерированному HTML (python:
   Title/Description/canonical, единственный H1, счётчики `page-price-card`,
   `compare__head-cell`/`data-label`, `--first`/`--last`, «ровно один
   `.section-lead` на `.section-head`» и **длина каждого `.section-lead` ≤ 250
   символов** (включая лид под H1) — для паритета прогони тот же скрипт
   по любой странице с теми же механизмами,
   FAQ против FAQPage 1:1, `json.loads` каждого JSON-LD). Браузер нужен только
   для HTTP 200, общего рендера и ширины 320 px. На Windows запускай python с
   `PYTHONIOENCODING=utf-8`, иначе печать кириллицы падает на cp1252.
7. **Скриншоты:** блоки скрыты анимацией `reveal` до скролла — перед
   полностраничным скриншотом добавь всем `.reveal` класс `is-visible`, иначе
   получишь ложные «пустоты»; повторы hero внизу длинного скриншота — артефакт
   склейки с fixed-фоном, не дефект страницы. Файлы скриншотов сохраняй в
   scratchpad, не в корень репозитория.

## Рабочий процесс

### 1. Семантическое ядро

Выполни [seo-semantic-core](../seo-semantic-core/SKILL.md) для целевого URL
целиком (обязательная проверка Вебмастера, Вордстат, кластеризация).
Результат: `seo-core/uslugi/<slug>.md`. Зафиксируй блокеры (индексация,
доступы) — они пойдут в финальный отчёт. SERP-проверку интента совмещай со
сбором выдачи для шага 2 (см. шпаргалку, п. 4).

### 2. Идеальная страница по конкурентам

Выполни [seo-ideal-page](../seo-ideal-page/SKILL.md) с ядром из шага 1
(выдача уже собрана на шаге 1 — переходи сразу к разбору страниц конкурентов).
Результат: `seo-pages/uslugi/<slug>.md` — готовый копирайт всех блоков с
обоснованиями, приложениями и чек-листом внедрения. Страницу этот шаг не меняет.

### 3. SEO-оптимизация контент-документа

Примени методику [seo-page-optimization](../seo-page-optimization/SKILL.md) к
**файлу** `seo-pages/uslugi/<slug>.md`, а не к странице: страница правится
только на шагах 4–5. Из скилла здесь используются разбор ядра, правила
распределения интентов по зонам (Title/H1/Description/лид/H2-H3/FAQ),
анти-переспам и проверка фактологии; внесение правок через генератор,
верификация страницы и отчёт откладываются на шаги 5–6.

Пройди по документу шага 2 и доведи его:

1. построй карту «подкластер/интент ядра → блок документа»; каждый P0/P1 покрыт
   блоком или явным объяснением, почему нет;
2. Title/H1/Description соответствуют правилам зон (P0 ближе к началу, цельные
   предложения, без перечислений ключей); проверь будущую уникальность
   Title/Description в пределах сайта (греп по `*/index.html`);
3. запросы «передать другим URL» не таргетируются — только перелинковка;
   исключённые запросы ядра в тексте не появились;
4. анти-переспам-проход по всему копирайту;
5. **проверка происхождения структуры:** структура документа сверху вниз — это
   будущая структура страницы, и у каждого её блока должно быть основание из
   конкурентного анализа или ядра (гигиенический минимум, лучшая находка,
   пустая ниша, подкластер). Каждая строка «Лучших находок» и «пустых ниш»
   приложения А либо реализована блоком/элементом, либо рядом записано, почему
   нет. Если ловишь себя на «возьму порядок секций как у ERP/CRM и адаптирую» —
   остановись и пересобери структуру от конкурентов: совпадение с существующими
   страницами сайта допустимо только как случайный итог анализа и требует
   обоснования каждого блока;
6. итоговую карту покрытия сохрани — она станет частью отчёта шага 6.

### 4. Сборка структуры страницы на блоках генератора

Структура берётся из контент-документа шага 3 — сверху вниз, как он написан.
Никакую существующую страницу за основу не бери:

1. Выпиши структуру документа: список блоков по порядку с их назначением
   (интент/находка, которые блок закрывает). Это и есть каркас страницы.
2. Каждому блоку подбери механизм из «Библиотеки блоков генератора»: ценовой
   блок → `{price-cards}`; сравнение или пары «проблема → результат» →
   `{compare}`; вводный блок с чек-листом сигналов → рендерер «О решении»;
   наборы карточек → бенто-ряды `SERVICE_BAND_ROWS`; вопросы → `SERVICE_FAQ`;
   остальное — секции `{section: …}` с прозой. Если для блока документа нет
   механизма — упрости до прозы/бенто или согласуй новую вёрстку с
   пользователем; не выкидывай блок молча.
3. Подключи slug к выбранным механизмам: `SEO_ENHANCED_SLUGS` (H1 в
   `.revolution` + Service/Offer/FAQPage — для SEO-цикла нужен практически
   всегда), и только те из `PRICING_IN_SIGNALS`, `SECTION_BODY_RENDERERS`,
   `SERVICE_BAND_ROWS`, `SERVICE_BAND_ICONS` (иконки переиспользуй из
   `images/why/`), `SERVICE_PRICING`, `SERVICE_FAQ`, `SEO_DESCRIPTIONS`,
   `SEO_KEYWORDS`, которые требует твоя структура. Синтаксис записей смотри
   у реализованных slug (греп), состав — нет.
4. `AI_DIRECTIONS_COPY`, `SERVICE_CTA`, stats и cases не трогай (служебные
   блоки).
5. Для отчёта шага 6 зафиксируй по каждому блоку: чем обоснован (конкурент /
   ядро / служебный) и каким механизмом реализован.

### 5. Наполнение контентом и генерация

1. Перепиши секцию целевой услуги в `docs/texts-usligi.md` контентом из
   `seo-pages/uslugi/<slug>.md`: `**Title:**` (без «— BUDGET SOFT», бренд

   с `{icon: …}`. FAQ из документа перенеси в `SERVICE_FAQ[slug]`, цену/сроки/

1. **Оформление блоков** — сверь сгенерированную страницу со справочником

   лид); если использован
   рендерер «О решении» — spec-карточки стоят внутри `erp-map__signals`; у
