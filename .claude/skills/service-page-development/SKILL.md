# Разработка страницы услуги по полному циклу

Проведи один существующий URL страницы услуги через весь конвейер: семантика →
эталонный контент по конкурентам → SEO-оптимизация контента → каркас блоков по
донору → готовая страница. Результат — обновлённая страница `uslugi/<slug>/` с
архитектурой блоков донорской страницы и контентом, обоснованным ядром и
анализом конкурентов, плюс артефакты каждого шага в `seo-core/`, `seo-pages/`,
`seo-optimization/`.

Скилл — оркестратор. Каждый шаг выполняется по своему скиллу, читай его перед
шагом:

1. [seo-semantic-core](../seo-semantic-core/SKILL.md) — семантическое ядро;
2. [seo-ideal-page](../seo-ideal-page/SKILL.md) — контент-документ по конкурентам;
3. [seo-page-optimization](../seo-page-optimization/SKILL.md) — методика
   оптимизации, конвейер генератора, верификация и отчёт;
4. [inner-page-blocks](../inner-page-blocks/SKILL.md) — справочник разметки
   блоков внутренних страниц (для проверки, что оформление не потерялось).

Эталон готового результата — [uslugi/razrabotka-erp-sistem/index.html](../../../uslugi/razrabotka-erp-sistem/index.html)
и его артефакты `seo-core/uslugi/razrabotka-erp-sistem.md`,
`seo-pages/uslugi/razrabotka-erp-sistem.md`, `seo-optimization/uslugi/razrabotka-erp-sistem.md`.

## Входные данные

1. **Целевая страница.** URL или путь вида `uslugi/<slug>/`. Проверь наличие в
   [urls.json](../../../urls.json). Если страницы там нет — это не разработка
   существующей страницы услуги, остановись и уточни у пользователя.
2. **Донор каркаса.** По умолчанию `/uslugi/razrabotka-erp-sistem/`. Другой донор —
   только если пользователь указал его явно.
3. **Готовые артефакты.** Если `seo-core/uslugi/<slug>.md` и/или
   `seo-pages/uslugi/<slug>.md` уже существуют и свежие (проверь дату анализа и
   совпадение Title/H1 с текущей страницей) — не пересобирай их молча: сообщи,
   что нашёл, и предложи продолжить с существующими либо пересобрать.
4. **Доступы.** Шаг 1 требует авторизации в Яндекс Вебмастере, шаги 1–2 —
   браузера (Playwright MCP) для Вордстата и выдачи. Блокировки доступов
   обрабатывай по правилам соответствующих скиллов, не имитируй данные.

## Неподменяемые правила

- **URL страницы неизменен** (правило CLAUDE.md): не переименовывай директорию,
  не трогай путь `uslugi/<slug>/index.html`. Меняется только содержимое.
- **Четыре блока не копируются с донора: Отрасли, Цифры, Портфолио, Обсудить
  проект.** На целевой странице они остаются свои. В терминах разметки и
  генератора это: «Отрасли» — `section.ai-directions`
  (`AI_DIRECTIONS_COPY[slug]`), «Цифры» — `section.stats`, «Портфолио» —
  `section.cases`, «Обсудить проект» — `section.cta` (`SERVICE_CTA[slug]`).
  Эти секции генератор выводит на каждой странице услуги автоматически — их
  словари для целевого slug не переписывай текстами донора. Улучшать их **свой**
  текст можно только когда этого требует ядро/контент-документ, а не «для
  симметрии с донором».
- **Правь источники генератора, а не сгенерированный HTML.** Почти весь HTML
  страницы услуги перезаписывается запуском `python _generate_pages.py`; правка
  `index.html` напрямую будет потеряна. Полная таблица «что менять → где
  источник» — в seo-page-optimization, раздел «Конвейер контента этого проекта».
- **Порядок шагов не меняй и шаги не пропускай.** Контент без ядра и анализа
  конкурентов — не результат этого скилла. Исключение — готовые свежие
  артефакты (см. «Входные данные», п. 3).
- **Факты компании — только с сайта или от пользователя.** Цены, сроки, кейсы,
  гарантии не выдумывай; несогласованные цифры несут маркер `[≈ согласовать]`
  из seo-ideal-page — перед публикацией их надо согласовать или убрать.
- **Не обещай позиций.** Итог — обоснованная страница, а не гарантия топа.

## Каркас донора в терминах генератора

«Скопировать всё кроме четырёх блоков» означает: дать целевому slug ту же
обвязку генератора, что у донора. У ERP-донора это:

| Элемент каркаса | Где включается |
|---|---|
| H1 в секции `.revolution` (а не в hero) + Service/Offer/FAQPage JSON-LD | `SEO_ENHANCED_SLUGS[slug]` в [_generate_pages.py](../../../_generate_pages.py) (`service_type`, `low_price`) |
| Секция «О решении» с картой `erp-map` (заголовок + лид + подзаголовок «когда оправдано» + чек-лист сигналов + иллюстрация) | `SECTION_BODY_RENDERERS[(slug, "О решении")] = make_about_body_renderer(...)` + секция `{section: О решении}` в MD |
| Карточки «Цена / Сроки / Формат оплаты» внутри `erp-map__signals` | `SERVICE_PRICING[slug]` + slug в `PRICING_IN_SIGNALS` |
| Пакеты стоимости карточками `.page-price-cards` | маркер `{price-cards}` перед таблицей в MD |
| Сравнительная таблица `.compare--service` | маркер `{compare}` перед таблицей в MD |
| Ряды бенто-карточек по два в ширину | `SERVICE_BAND_ROWS[slug]` — группы заголовков `###` из MD |
| Иконки карточек «Почему мы» (`images/why/*.png`) | `SERVICE_BAND_ICONS` — по точному тексту заголовка карточки |
| Блок «Частые вопросы» + FAQPage-разметка 1:1 | `SERVICE_FAQ[slug]` |
| Title / Description / Keywords | `SEO_TITLES[slug]`, `SEO_DESCRIPTIONS[slug]`, `SEO_KEYWORDS[slug]` |
| Структура и текст секций | секция услуги в [docs/texts-usligi.md](../../../docs/texts-usligi.md): `**Title:**`, `**H1:**`, лид, затем блоки `{section: <Бровь>}` + `### Заголовок`; `{icon: имя}` в пунктах списков модулей |

Состав секций MD у донора: `О решении` → `Стоимость` (`{price-cards}`) →
`Сравнение подходов` (`{compare}`) → `Процесс и технологии` (этапы, модули,
стек, интеграции) → `Почему мы` (4 карточки-преимущества). Второй реализованный
по этой схеме slug — `razrabotka-crm-sistem`: используй его записи в словарях
как образец подключения нового slug.

## Рабочий процесс

### 1. Семантическое ядро

Выполни [seo-semantic-core](../seo-semantic-core/SKILL.md) для целевого URL
целиком (обязательная проверка Вебмастера, Вордстат, кластеризация).
Результат: `seo-core/uslugi/<slug>.md`. Зафиксируй блокеры (индексация,
доступы) — они пойдут в финальный отчёт.

### 2. Идеальная страница по конкурентам

Выполни [seo-ideal-page](../seo-ideal-page/SKILL.md) с ядром из шага 1.
Результат: `seo-pages/uslugi/<slug>.md` — готовый копирайт всех блоков с
обоснованиями, приложениями и чек-листом внедрения. Страницу этот шаг не меняет.

### 3. SEO-оптимизация контент-документа

Примени методику [seo-page-optimization](../seo-page-optimization/SKILL.md) к
**файлу** `seo-pages/uslugi/<slug>.md`, а не к странице: страница правится
только на шагах 4–5. Из скилла здесь используются разбор ядра, правила
распределения интентов по зонам (Title/H1/Description/лид/H2-H3/FAQ),
анти-переспам и проверка фактологии; внесение правок через генератор,
верификация страницы и отчёт откладываются на шаги 5–6.

Пройди по документу шага 2 и доведи его:

1. построй карту «подкластер/интент ядра → блок документа»; каждый P0/P1 покрыт
   блоком или явным объяснением, почему нет;
2. Title/H1/Description соответствуют правилам зон (P0 ближе к началу, цельные
   предложения, без перечислений ключей); проверь будущую уникальность
   Title/Description в пределах сайта (греп по `*/index.html`);
3. запросы «передать другим URL» не таргетируются — только перелинковка;
   исключённые запросы ядра в тексте не появились;
4. анти-переспам-проход по всему копирайту;
5. итоговую карту покрытия сохрани — она станет частью отчёта шага 6.

### 4. Каркас целевой страницы по донору

Перенеси архитектуру донора на целевой slug, кроме четырёх исключённых блоков:

1. Прочитай секцию донора в `docs/texts-usligi.md` и его записи во всех
   словарях из таблицы выше (греп по slug донора).
2. Подключи целевой slug к тем же механизмам: добавь записи в
   `SEO_ENHANCED_SLUGS`, `PRICING_IN_SIGNALS`, `SECTION_BODY_RENDERERS`
   (свой `make_about_body_renderer` с формулировками целевой услуги),
   `SERVICE_BAND_ROWS`, при новых заголовках карточек — `SERVICE_BAND_ICONS`
   (иконки переиспользуй из `images/why/`), `SERVICE_PRICING`, `SERVICE_FAQ`,
   `SEO_TITLES`, `SEO_DESCRIPTIONS`, `SEO_KEYWORDS`.
3. Каркас — это **типы и порядок блоков**, а не их количество и не тексты:
   состав секций диктует контент-документ шага 3. Если документ обосновывает
   отсутствие какого-то блока донора (например, сравнивать не с чем) или
   дополнительный бенто-блок — следуй документу, отклонение зафиксируй для
   отчёта.
4. `AI_DIRECTIONS_COPY`, `SERVICE_CTA`, stats и cases не трогай (исключённые
   блоки).

### 5. Наполнение контентом и генерация

1. Перепиши секцию целевой услуги в `docs/texts-usligi.md` контентом из
   `seo-pages/uslugi/<slug>.md`: `**Title:**` (без «— BUDGET SOFT», бренд
   дописывает генератор), `**H1:**`, лид, блоки `{section: …}` в структуре
   каркаса, таблицы с маркерами `{price-cards}` и `{compare}`, списки модулей с
   `{icon: …}`. FAQ из документа перенеси в `SERVICE_FAQ[slug]`, цену/сроки/
   оплату — в `SERVICE_PRICING[slug]`.
2. Секцию «О решении» пиши по контракту `make_about_body_renderer`: вводный
   абзац заканчивается фразой-заголовком чек-листа с двоеточием, затем список
   сигналов — иначе рендерер не сработает и блок останется прозой.
3. Запусти `python _generate_pages.py` из корня.
4. Проверь `git diff`: изменилось только содержимое целевой страницы и общих
   артефактов генерации (sitemap `lastmod` может сдвинуться у соседних страниц —
   это не контентные правки); пути не переименованы; чужие страницы не
   изменились по смыслу.
5. Обнови поле `title` записи целевой страницы в `urls.json` (содержимое, не URL).

### 6. Верификация и отчёт

1. **Оформление блоков** — сверь сгенерированную страницу с
   [inner-page-blocks](../inner-page-blocks/SKILL.md) и донором: после каждого
   `.section-title` ровно один `.section-lead`; spec-карточки стоят внутри
   `erp-map__signals`; у `{compare}`-таблицы есть `compare__head` и `data-label`
   на ячейках; бенто-ряды получили `--first`/`--last`; FAQ — аккордеон
   `page-faq` с общим `name`.
2. **Он-пейдж-проверки из seo-page-optimization, шаг 5:** Title/Description/H1
   на месте и уникальны, JSON-LD парсится как JSON, видимый FAQ совпадает с
   разметкой 1:1, canonical и robots не сломаны.
3. Подними локальный сервер (`python -m http.server 8000`), открой целевую
   страницу: HTTP 200, рендер без ошибок, читаемость на ширине 320 px.
4. Создай отчёт `seo-optimization/uslugi/<slug>.md` по шаблону
   seo-page-optimization (карта покрытия до/после — из шага 3, таблица правок,
   блокеры, план контроля).
5. Индексацию (переобход в Вебмастере, IndexNow) предложи по правилам
   seo-page-optimization, шаг 6 — только после подтверждённого деплоя.

## Чек-лист качества

- [ ] URL и директория целевой страницы не изменены; `urls.json` — только поле `title`.
- [ ] Все три артефакта существуют: `seo-core/…`, `seo-pages/…`, `seo-optimization/…`.
- [ ] Каждый P0/P1 ядра покрыт блоком страницы или объяснением в отчёте.
- [ ] Блоки Отрасли, Цифры, Портфолио, Обсудить проект не скопированы с донора.
- [ ] Правки внесены в источники генератора; `python _generate_pages.py` выполнен; diff проверен.
- [ ] Оформление всех блоков совпадает с донором/справочником; ни один блок не «расклеился» в прозу.
- [ ] Видимый FAQ и FAQPage-разметка совпадают 1:1; JSON-LD валиден.
- [ ] Ни один факт не выдуман; маркеры `[≈ согласовать]` согласованы или убраны с фиксацией в отчёте.
- [ ] Анти-переспам-проход по итоговому видимому тексту выполнен.
- [ ] Страница проверена на локальном сервере, включая ширину 320 px.

## Отчёт пользователю

Сообщи: пути к трём артефактам и целевой странице; итоговые Title/H1/Description
(было → стало); сколько P0/P1-интентов покрыто и что осталось; какие блоки
каркаса перенесены с донора и какие отклонения допущены по контент-документу;
несогласованные цифры; блокеры индексации/Вебмастера; что сделано или отложено
по индексации и когда сравнивать метрики (ориентир 4–6 недель после переобхода).
