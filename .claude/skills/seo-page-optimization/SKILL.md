# Он-пейдж SEO-оптимизация страницы под Яндекс по готовому ядру

Возьми готовое семантическое ядро одной страницы и внедри его в контент этой страницы:
осмысленными правками зон документа, а не вписыванием ключей. Результат — изменённая
страница плюс отчёт `seo-optimization/<путь-страницы>.md` с картой покрытия
интентов «до/после».

Перед работой прочитай:

- [references/metodika.md](references/metodika.md) — зоны документа, правила распределения
  запросов, анти-фильтры, коммерческие и поведенческие факторы, индексация;
- [references/shablon-otcheta.md](references/shablon-otcheta.md) — обязательный формат отчёта.

Парный скилл: `seo-semantic-core` собирает ядро и создаёт `seo-core/<путь>.md`;
этот скилл — следующий шаг конвейера, он ядро **потребляет**, а не собирает.

## Входные данные

1. **URL или путь страницы.** Проверь наличие пути в [urls.json](../../../urls.json).
   Если страница не указана и её нельзя однозначно определить — спроси.
2. **Файл семантического ядра.** По умолчанию `seo-core/<путь-страницы>.md` (формат
   скилла seo-semantic-core). Если пользователь дал другой файл — используй его, но
   сопоставь колонки с ожидаемыми (запрос, приоритет, подкластер, элемент страницы).
   Если файла нет — не собирай семантику на лету: предложи сначала выполнить
   seo-semantic-core и остановись.
3. **Свежесть ядра.** Сравни «Дату анализа» с текущей датой и проверь, не изменились ли
   оффер, страница и интент выдачи. Возраст 3–6 месяцев — сигнал перепроверить ключевые
   решения, а не автоматический стоп. Остановись только если пересбор может изменить
   целевой URL, основной интент или согласованный объём работ.
4. **Границы правок.** По умолчанию меняй только источники целевой страницы. Входящие
   ссылки с других страниц сначала перечисли в плане; меняй их, если это явно входит в
   запрос пользователя. Новые страницы/статьи, цены, кейсы и оффер — только по явному
   запросу.

## Неподменяемые правила

- **URL страницы неизменен.** Не переименовывай директорию, не трогай `<dir>/index.html`
  как путь. Менять можно только содержимое (правило CLAUDE.md).
- **Правь источник, а не сгенерированный HTML.** Прежде чем менять любую страницу,
  определи, откуда берётся её контент (см. «Конвейер контента» ниже). Правка
  сгенерированного `index.html` напрямую будет потеряна при следующей генерации.
- **Не выдумывай факты.** Цены, сроки, кейсы, гарантии, интеграции, регалии — только
  из существующего контента сайта или от пользователя. Если рекомендация ядра требует
  факта, которого нет, — зафиксируй пробел в отчёте и спроси, не сочиняй.
- **Контентные правки привязывай к ядру или качеству страницы.** У изменения должно
  быть обоснование: запрос/интент/подкластер/рекомендация ядра либо выявленная проблема
  индексируемости, достоверности, полезности или удобства. Декоративные правки вне
  скоупа.
- **Покрытие оценивай по интенту и смысловому ответу.** Леммы используй как вторичную
  проверку. Точная словоформа нужна только там, где естественна; перечни запросов и
  искусственное раздувание текста запрещены.
- **Запросы со статусом «передать другому URL» не таргетируй.** Для них допустима
  только перелинковка на целевой URL из таблицы передачи ядра.
- **Блокеры ядра не игнорируй.** Если хост не добавлен в Вебмастер или страница не в
  индексе, вынеси это первым пунктом отчёта. Уточняй, что недоступна наблюдаемость через
  Вебмастер, а не что эффект принципиально невозможно измерить другими средствами.
- **FAQ и разметка совпадают 1:1.** Видимый текст вопросов/ответов и FAQPage-разметка
  не должны расходиться.
- **Не обещай позиций и не называй длину Title/Description «правилом Яндекса»:**
  официально длина ограничена шириной экрана, а не числом символов.

## Конвейер контента этого проекта

Почти все страницы сайта генерируются скриптом [_generate_pages.py](../../../_generate_pages.py)
(запуск: `python _generate_pages.py` из корня). Источники по типам страниц:

| Что менять | Где источник |
|---|---|
| Title и H1 страницы услуги `uslugi/<slug>/` | `docs/texts-usligi.md`, секция услуги: строки `**Title:**` и `**H1:**`; в `Title` бренд не добавляй — генератор сам дописывает `— BUDGET SOFT` |
| Основной текст страницы услуги | тело той же секции `docs/texts-usligi.md` (markdown) |
| Meta description услуги | `SEO_DESCRIPTIONS[slug]` в `_generate_pages.py`; без переопределения берётся первый абзац текста, обрезанный до 157 символов — для оптимизируемой страницы всегда задавай явное переопределение цельным предложением |
| Meta keywords (точечно, без спама) | `SEO_KEYWORDS[slug]` в `_generate_pages.py` |
| Блок «Частые вопросы» + FAQPage-разметка | `SERVICE_FAQ[slug]` в `_generate_pages.py` (виден на странице и дублируется в JSON-LD 1:1) |
| Расширенная Service/Offer/FAQPage-разметка | `SEO_ENHANCED_SLUGS[slug]` в `_generate_pages.py` (`service_type`, `low_price`) |
| Цена/сроки/оплата услуги | `SERVICE_PRICING[slug]` в `_generate_pages.py` |
| Страницы верхнего уровня (`etapy`, `sroki`, `garantii`, `stoimost`, `importozameshchenie`, `o-nas`, …) | словари `pages_depth1` и функции `content_*` в `main()` `_generate_pages.py` |
| Главная `index.html` | `update_index_html()` / `inject_home_seo()` в `_generate_pages.py` (частично правится на месте) |
| sitemap.xml, robots.txt | генерируются автоматически из `urls.json` при запуске скрипта — руками не править |

Перед правкой найди slug страницы грепом по `_generate_pages.py` и `docs/texts-usligi.md`.
Если страница нигде в генераторе не упоминается — только тогда правь её HTML напрямую.
Канонический хост — `https://www.budget-soft.ru`; Метрика вставляется автоматически.

В `urls.json` у записей есть поле `title`: после смены Title синхронизируй **содержимое**
записи (URL не трогай).

## Рабочий процесс

### 1. Валидация входных данных

1. Найди путь страницы в `urls.json` и локальный `<путь>/index.html`.
2. Прочитай `seo-core/<путь>.md` целиком. Проверь дату анализа, регион, блокеры.
3. Если ядро ссылается на устаревшую версию страницы (Title/H1 в ядре не совпадают с
   текущими), зафиксируй расхождение: возможно, часть рекомендаций уже внедрена.
4. Выполни preflight целевого URL: HTTP 200, self-canonical, отсутствие запрета в
   `robots.txt` и `meta robots`, индексируемый основной контент, наличие внутренних
   ссылок, корректность мобильного отображения. Если доступен Вебмастер, используй
   «Проверку страницы» и зафиксируй статус в поиске и представление страницы роботу.
5. Разделяй блокеры выполнения и наблюдаемости. Запрет индексации или неверный
   canonical блокирует продвижение URL; отсутствие доступа к Вебмастеру блокирует
   только часть диагностики и статистики.

### 2. Разбор ядра

Извлеки из файла ядра структурированно:

- P0/P1/P2-запросы по подкластерам и назначенные им «элементы страницы»;
- рекомендации раздела «Рекомендации по странице» — это основной план работ;
- таблицу «Передать другим URL» — план перелинковки;
- «Кандидатов на отдельный материал» — не внедрять, упомянуть в отчёте;
- исключённые запросы — не использовать нигде;
- «Текущую видимость URL» — базовая линия для мониторинга;
- блокеры и ограничения источников.

### 3. Аудит текущей страницы

1. Определи источник контента по таблице конвейера (грепом по slug).
2. Составь карту зон документа: Title, Description, H1, H2/H3, лид и первый экран,
   основной текст по блокам, цена, FAQ, кейсы/портфолио, внутренние ссылки (входящие
   и исходящие), JSON-LD, alt изображений.
3. Построй карту «подкластер/интент → назначенная зона → фактический ответ» **до
   правок**. Для P0/P1 укажи подтверждающий фрагмент, статус `покрыт / частично /
   не покрыт / передан` и только затем проверь леммы. Варианты одного интента не
   считай отдельными контентными задачами.
4. Зафиксируй непокрытые задачи и отсутствующие блоки (например, нет объяснения цены
   или ответа про сроки), а не только отсутствующие словоформы.
5. Оцени страницу по ЭПОС: экспертность, полезность, оригинальность и
   содержательность. Не предлагай общие фразы вместо доказательств, примеров и
   практически полезной информации.

### 4. Проектирование правок

Для каждой зоны подготовь «было → станет → обоснование (строка ядра)». Правила
распределения — в [references/metodika.md](references/metodika.md), кратко:

- **Title** — главный P0-макроинтент ближе к началу, читаемое описание страницы, без
  перечислений и повторов. В `docs/texts-usligi.md` бренд не добавляй: его добавляет
  генератор.
- **H1** — называет предложение словами основного P0-запроса; не обязан дублировать
  Title дословно.
- **Description** — цельные предложения, самое важное в начале, полезные факты (цена
  «от…», сроки, формат работы); уникален в пределах сайта.
- **Лид/первый экран** — интент P0 и ценностное предложение ясны без прокрутки;
  сервисные формулировки («услуги по…») — если ядро подтверждает такой спрос.
- **H2/H3** — отражают только подкластеры, для которых есть содержательный блок;
  вопросный спрос можно раскрывать естественным вопросным заголовком.
- **Текст блоков** — раскрывает подтверждённые подтемы; синонимы и варианты из P1/P2
  используй только когда они естественно помогают изложению, без нормы повторений.
- **FAQ** — формулируй вопросы естественно, сохраняя интент подтверждённого спроса;
  ответы должны содержать только реальные факты. Синхронизируй FAQPage-разметку.
- **Перелинковка** — исходящие ссылки по таблице передачи ядра (анкор по смыслу);
  входящие — с релевантных страниц сайта на целевую.
- **Разметка** — BreadcrumbList/Organization/Service/Offer/FAQPage должны быть валидны
  и соответствовать видимому контенту. Не обещай расширенный сниппет: Яндекс
  документирует его только для отдельных поддерживаемых сценариев; `Product + Offer`
  для товара нельзя автоматически переносить на `Service + Offer`.

Обязательные проверки проекта правок:

1. **Уникальность Title/Description** — грепни `<title>` и `meta name="description"`
   по всем `*/index.html`: новые значения не должны дублировать другие страницы.
2. **Анти-переспам-проход** — прочитай итоговый текст как редактор: нет перечней
   запросов, нет неестественных конструкций «ключ ради ключа», ни одна лемма не
   доминирует в абзаце. Сомневаешься — упрощай формулировку, покрытие леммой важнее
   точной формы.
3. **Фактология** — каждое числовое или продуктовое утверждение имеет источник на
   сайте или подтверждено пользователем.

### 5. Внесение правок и верификация

1. Внеси изменения в файлы-источники (см. конвейер), затем запусти
   `python _generate_pages.py`.
2. Проверь `git diff`: изменилось только содержимое; ни одна директория/путь не
   переименованы; чужие страницы не изменились непреднамеренно. Учти, что текущий
   генератор считает `sitemap.xml:lastmod` по mtime сгенерированного HTML и может
   обновить даты нескольких страниц даже без смысловых изменений; зафиксируй это
   отдельно, не выдавай массовую смену `lastmod` за изменение контента.
3. Проверь результат в сгенерированном HTML: Title/Description/H1/FAQ на месте,
   JSON-LD валиден (распарси его как JSON), видимый FAQ совпадает с разметкой 1:1.
4. Подними локальный сервер (например, `python -m http.server 8000`) и убедись, что
   страница отдаётся с HTTP 200, рендерится без ошибок и читается на ширине 320 px.
5. Повтори preflight по итоговому HTML: canonical, robots, H1, внутренние ссылки и
   основной контент не потеряны.
6. Обнови поле `title` записи страницы в `urls.json` (содержимое, не URL).

### 6. Индексация изменений

После публикации (если пользователь подтвердил деплой) предложи и при возможности
выполни:

1. **Переобход в Вебмастере**: Индексирование → «Переобход страниц» → целевой URL.
   Дневной лимит индивидуален. Робот обычно посещает URL не позднее трёх дней, но
   обновление данных в поиске может занять до двух недель. Авторизация — как в
   seo-semantic-core: открой Вебмастер в браузере и попроси пользователя войти самому;
   логин/пароль в чате не запрашивай.
2. **IndexNow** — `GET https://yandex.com/indexnow?url=...&key=...` при наличии
   ключевого файла на сайте; индексация не гарантируется.
3. Проверь раздел «Диагностика» Вебмастера на предмет нарушений после переиндексации
   (при плановом контроле, не сразу).

Если сайт ещё не задеплоен или в ядре есть блокер Вебмастера — зафиксируй эти шаги
в отчёте как отложенные, не имитируй их выполнение.

### 7. Отчёт

Создай `seo-optimization/<путь-страницы-без-index.html>.md` по
[references/shablon-otcheta.md](references/shablon-otcheta.md): scope, блокеры, карта
покрытия до/после, таблица правок с обоснованиями, непокрытые запросы с причинами,
статус индексации, базовая линия и план контроля.

### 8. Чек-лист качества

- [ ] URL и директория страницы не изменены; правки — только в содержимом.
- [ ] Правки внесены в источник генератора, `python _generate_pages.py` выполнен,
  diff проверен.
- [ ] Каждая контентная правка обоснована ядром или выявленной проблемой качества.
- [ ] Главный P0-макроинтент понятен из Title, H1 и первого экрана; остальные P0
  раскрыты в назначенных ядром зонах, исключения объяснены.
- [ ] Запросы «передать другим URL» не таргетируются; перелинковка по таблице ядра.
- [ ] Исключённые запросы ядра не появились в тексте.
- [ ] Title/Description уникальны в пределах сайта; без перечислений ключей.
- [ ] Видимый FAQ и FAQPage-разметка совпадают 1:1; JSON-LD валиден.
- [ ] Ни один факт не выдуман; пробелы фактологии зафиксированы в отчёте.
- [ ] Анти-переспам-проход выполнен по итоговому видимому тексту.
- [ ] Preflight до/после пройден: 200, canonical, robots, индексируемый контент,
  внутренние ссылки и мобильное отображение.
- [ ] `urls.json` синхронизирован (title записи), sitemap перегенерирован.
- [ ] Блокеры из ядра вынесены в отчёт первым пунктом.
- [ ] Базовая линия и план контроля записаны; позиции не обещаны.

## Отчёт пользователю

Сообщи: путь к отчёту, список изменённых зон (было → стало для Title/H1/Description),
сколько P0/P1-интентов покрыто и что осталось непокрытым и почему, какие блокеры
мешают увидеть эффект, что сделано по индексации и когда сравнивать метрики
(сопоставимые периоды после переобхода и обновления данных; ориентир 4–6 недель, для
низкого спроса — до накопления достаточных показов). Если Вебмастер показывает данные
«Видимости сайта в Алисе AI», добавь SoV как отдельную метрику. Напомни, что
изменение позиций зависит также от сезонности, конкурентов и апдейтов алгоритма — не
только от правок.
