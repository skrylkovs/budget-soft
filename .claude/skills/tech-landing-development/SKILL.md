# Разработка стекового лендинга технологии под ключ

> `C:\Users\skryl\PhpstormProjects\prototype-main`.** На диске `F:` лежит
> полная копия репо (включая `.claude/skills`) — Skill-тул может отдать
> f:\-base directory и молча увести всю сессию в копию (случилось 16.07.2026;
> см. память `f-drive-not-repo-alias`). Все пути этого скилла резолви от C:.

> **Статус раздела (16.07.2026):** B0 выполнен, `/tehnologii/` и
> `/tehnologii/laravel/` существуют. Laravel прошёл конвейер целиком (ядро,
> документ, страница, перелинковка, отчёт) — он и есть живой эталон. Новый
> стек = контент + одна запись реестра: см. «Онбординг нового стека» ниже.

Проведи один стек (laravel, python, react, …) через весь конвейер: слаг →
облегчённое ядро → контент-документ по эталону раздела → SEO-оптимизация
документа → сборка и генерация → хаб, перелинковка, верификация, отчёт.
Результат — **новая** страница `/tehnologii/<slug>/` и артефакты каждого шага
в `seo-core/tehnologii/`, `seo-pages/tehnologii/`, `seo-optimization/tehnologii/`.

Скилл построен на [service-page-development](../service-page-development/SKILL.md),
но четыре его основы **инвертированы** — не переноси привычки оттуда молча:

| | Страница услуги (service-page-development) | Стековый лендинг (этот скилл) |
|---|---|---|
| URL | существующий, неизменен | **новый**, создаётся навсегда (слаг согласуется до создания) |
| Структура | собирается с нуля из контент-документа | **фиксированный каркас эталона** на весь раздел; конкуренты дают наполнение |
| Семантика | полный цикл с историей URL в Вебмастере | облегчённый цикл: истории URL нет по определению |
| Блоки | служебные spec-cards/compare--service | эти два — **запрещённые маркеры**; служебные «Отрасли», «Цифры»+«Портфолио» и CTA вставляет `build_tehnologii_page`/`render_page` |

Скилл — оркестратор. Шаги выполняются по своим скиллам (читай каждый перед
шагом, включая их `references/*`):

1. [seo-semantic-core](../seo-semantic-core/SKILL.md) — ядро (адаптация «нового URL» — ниже);
2. [seo-ideal-page](../seo-ideal-page/SKILL.md) — контент-документ (с переопределениями шага 3);
3. [seo-page-optimization](../seo-page-optimization/SKILL.md) — методика зон, анти-переспам, отчёт;
4. [inner-page-blocks](../inner-page-blocks/SKILL.md) — справочник **нейтральных** блоков
   (шапка секции, бенто, page-faq); его живой пример — страница услуги ERP,
   состав с неё не копировать.

**Эталон раздела — [seo-pages/tehnologii/laravel.md](../../../seo-pages/tehnologii/laravel.md):**
источник истины по каркасу (блоки 1–9 + FAQ + финальный CTA; в таблице
каркаса ниже они пронумерованы позициями 1–11), конвенциям документа
(обоснования блоков, `[≈ согласовать]`, приложения А/Б/В) и запретам.
Раздел 0 эталона объясняет, чем лендинг отличается от страницы услуги, —
прочитай его целиком. При конфликте с п. 2 этапа B0
[плана](../../../docs/plan-novye-stranicy-uslug-i-stekovye-lendingi.md)
(там ещё «compare-таблица» и «прайс-карточки Цена/Сроки/Формат» — текст написан
до эталона) **прав эталон**.

## Входные данные

1. **Стек.** Слаг из плана трека B (волна B1: `python`, `react`, `flutter`,
   `nodejs`, `laravel`, `vue`; B3 — расширенный список). Если слага в плане
   нет — согласуй с пользователем **до** создания каталога: URL необратимы.
   Проверь `urls.json`: страницы `/tehnologii/<slug>/` там быть не должно
   (если есть — это уже не «создание», остановись и уточни).
2. **Эталон и план** (ссылки выше). Эталон обязателен к прочтению целиком.
3. **Готовые артефакты.** Если `seo-core/tehnologii/<slug>.md` и/или
   `seo-pages/tehnologii/<slug>.md` уже существуют — не пересобирай молча:
   сообщи и предложи продолжить с ними либо пересобрать. Для `laravel` весь
   комплект готов (ядро + документ + отчёт) — это живой эталон конвейера.
4. **Доступы.** Ядро требует авторизации в Яндекс Вебмастере и Вордстате
   (авторизуется пользователь; сессия Яндекса в общем браузерном профиле
   периодически слетает — попроси войти заново). Браузер Playwright — общий
   для всех сессий Claude: если занят чужой активной сессией, не воюй
   (протокол — память `playwright-mcp-shared-profile`). Блокировки
   обрабатывай по правилам суб-скиллов, данные не имитируй.

## Неподменяемые правила

- **URL создаётся навсегда.** Слаг согласован; в `urls.json` — только
  **добавление** записей (`count` увеличить на число добавленных), существующие
  записи и адреса не трогать (CLAUDE.md). Первый запуск добавляет два URL:
  хаб `/tehnologii/` и лендинг.
- **Каркас страницы фиксирован эталоном** (таблица ниже). Состав и порядок
  блоков не изобретай заново и не срисовывай со страниц услуг. Отклонение —
  только по данным выдачи конкретного стека, с обоснованием в контент-документе
  в конвенции эталона (интент / счёт конкурентов / пустая ниша).
- **Запрещённые маркеры страниц услуг:** `page-spec-cards` и
  `compare--service`. В MD-источнике **не используй маркер `{compare}`** —
  `render_compare` эмитит именно `compare--service`. Блок «Почему мы»
  карточками не собирается: доверие живёт в интро-панели первого экрана и
  кейсах. Блок «Отрасли» (`ai-directions`) с 16.07.2026 **разрешён и
  служебный** (решение пользователя, отменяет старый запрет эталона) — его
  добавляет `build_tehnologii_page` автоматически, см. следующий пункт.
- **Служебные блоки каркаса остаются свои:** `build_tehnologii_page`
  автоматически добавляет после контент-блоков секцию «Отрасли»
  (`render_ai_directions_block`, копия — `AI_DIRECTIONS_COPY["tehnologii-<slug>"]`,
  без ключа — дефолтная; для нового стека добавь копию рядом с laravel),
  затем `render_page` — «Цифры» + «Портфолио» (`partials/stats-cases.html`,
  `include_stats_cases=True`) и CTA — это реализация позиций 6 и 11 каркаса,
  в `docs/texts-tehnologii.md` они **не переносятся**. Партиал общий для
  всего сайта и фиксированный — не переписывай его и не дублируй кейсы
  контент-блоком; текст финального CTA задаётся полем `**CTA:**` секции MD
  (`заголовок :: кнопка :: подпись`, стек — в заголовке).
- **Правь источники генератора, не сгенерированный HTML.** После B0 источник —
  `docs/texts-tehnologii.md` и tehnologii-структуры в
  [_generate_pages.py](../../../_generate_pages.py). Оговорку
  seo-page-optimization «страница не упоминается в генераторе → правь HTML»
  не применяй: B0 обязателен, прямая правка HTML будет перезаписана прогоном.
- **Факты компании — только с сайта или от пользователя;** несогласованные
  цифры — с маркером `[≈ согласовать]`, сводка в приложении В документа и в
  отчёте. Грабли фактуры (проверены на эталоне):
  - 1 400 ₽/час — **нижняя ставка сетки аутстаффинга, принадлежит
    QA-инженеру**; разработчик Middle+ — от 1 900 ₽/час. Не подавай 1 400 ₽
    как ставку разработки; связка любой ставки со стеком — `[≈ согласовать]`;
  - кейсы портфолио **не тегированы стеками** — подача «веб-проекты
    BUDGET SOFT», без приписывания стека, пока бизнес не подтвердит;
  - срок гарантийного периода в месяцах на сайте нигде не назван —
    не публикуй («фиксируем в договоре»);
  - отзывов и профилей команды на сайте нет — блоки «отзывы»/«команда»
    не собирать (отклонённые ходы, см. приложение А эталона).
- **Ссылки — только на URL из `urls.json`.** Будущие `/tehnologii/*` упоминай
  текстом без ссылок (предохранитель эталона). При выходе нового лендинга
  сними предохранители на уже опубликованных страницах раздела (взаимные
  ссылки в блоке «подходит/не подходит») и добавь стек в хаб.
- **Целевой объём видимого контента — заметно короче страницы услуги;**
  дефолт 7–9 тыс. знаков (диапазон сильной семёрки конкурентов эталона;
  медиана всех 11 — 6,4 тыс.). На каждом стеке сверяй с медианой своей выдачи
  и при существенном расхождении бери данные стека, зафиксировав решение в
  документе. Этапы — только старт (3 шага), полный цикл ссылкой на `/etapy/`.
- **Запросы «<стек> разработчики»** и прочие с карьерной органикой в мета не
  таргетируй — коммерческий остаток интента снимается блоком «инженеры в
  команду» (так в эталоне, приложение Б).
- **Не обещай позиций.**

## Каркас стекового лендинга (обязателен для раздела)

Одиннадцать позиций эталона сверху вниз. «Стек-специфично» — то, что на каждом
стеке выводится из выдачи и ядра; остальное — инвариант шаблона:

| № | Блок | Механика источника | Стек-специфично |
|---|---|---|---|
| 1 | Первый экран — живёт в секции `.revolution` (штатный hero выше не трогать: его заголовок общесайтовый и фиксированный, он даёт кнопки CTA [Обсудить проект] + `second_btn` [Рассчитать стоимость → /stoimost/]): H1 «Разработка на X» через `title_html`, подзаголовок-оффер — лид (≤ 250 знаков), **интро-панель `.tech-intro`** — тёмная витрина: слева факты 2×2 крупными цифрами, справа картинка стека | поле `**Facts:**` («цифра :: подпись» через ·) + `intro_image`/`intro_alt` в `TECH_META`; картинка `images/tech/<slug>-hero.svg` — SVG-мокап редактора, генерируется `_tech_hero_svg.py` (добавь спеку стека в словарь `S`) | оффер — против пустой ниши выдачи стека; факты общесайтовые |
| 2 | **Матрица решений** «Что строим \| Срок \| Цена \| Подробнее» — ядро шаблона; 5–7 строк, каждая ведёт на страницу `/uslugi/`; цены и сроки = цены этих страниц услуг | обычная MD-таблица → `.page-service-table` | состав строк — типы проектов, релевантные стеку |

| 7 | «Достался проект на X?»: аудит кода, спасение проектов, модернизация legacy, техподдержка — кликабельные карточки услуг с иконками | `{rescue-cards}` (пункты `- **[Название](/uslugi/…/)** — описание`; иконка по slug из `RESCUE_ICONS`, новым услугам добавь иконку) | формулировки болей стека; тизер миграций B2 |

**Дизайн-система блоков раздела (утверждена пользователем 16–17.07.2026):**

## Рабочий процесс

### 1. Проверка входа

Убедись, что работаешь в C:-репозитории (предупреждение в шапке скилла).
Слаг (согласован? есть в плане?), отсутствие URL в `urls.json`, наличие и
свежесть готовых артефактов, целостность генератора
(`python -c "import _generate_pages"`). Любую неопределённость со слагом
решай с пользователем до создания файлов.

### 2. Семантическое ядро (облегчённый цикл)

Выполни [seo-semantic-core](../seo-semantic-core/SKILL.md) для будущего URL
`/tehnologii/<slug>/` с адаптацией нового URL:

- роль страницы бери из эталона и плана трека B, а не из чтения страницы
  (её нет); маркерные формулировки — из п. 3 шпаргалки;
- Вебмастер — по п. 1 шпаргалки, Вордстат/подсказки — по п. 2; дальше штатно:
  операторные проверки + SERP-проверка интента (совмести со сбором выдачи
  для шага 3); образец готового ядра — `seo-core/tehnologii/laravel.md`;
- приоритизация без критерия «текущая видимость URL» (везде «н/д — страница
  не создана», не выдумывай значения);
- сверка подкластеров с `urls.json` обязательна — это защита от
  каннибализации с `/uslugi/` (п. 5 шпаргалки).

Результат: `seo-core/tehnologii/<slug>.md`. Блокеры (хост, доступы) — в
финальный отчёт.

### 3. Контент-документ по эталону

Если `seo-pages/tehnologii/<slug>.md` существует (laravel — это сам эталон) —
проверь свежесть, актуализируй по ядру шага 2 (приложение Б эталона прямо
этого требует) и иди дальше. Актуализация эталона меняет только
стек-специфичное наполнение, FAQ и приложения; раздел 0, состав и порядок
блоков и конвенции документа — контракт всего раздела, их менять только с
явного согласования пользователя. Иначе выполни
[seo-ideal-page](../seo-ideal-page/SKILL.md) с переопределениями:

1. маркерные формулировки передай явно из ядра — fallback «из Title/H1
   страницы» для нового URL невозможен;
2. **состав блоков не с нуля:** каркас фиксирован эталоном раздела — это
   осознанное переопределение правила «структуру диктуют конкуренты» (эталон
   и есть продукт того правила, зафиксированный на весь раздел). Конкуренты
   дают стек-специфичное наполнение: типы решений матрицы, альтернативу
   сравнения, инструменты экосистемы, FAQ-формулировки, ходы для Title,
   ориентир объёма — и локальные отклонения от каркаса с обоснованием;
3. разбор конкурентов — облегчённый: 6–8 страниц из выдачи стека (порог «≥ 6
   либо объяснение» из чек-листа seo-ideal-page сохраняется; эталону
   потребовалось 11 — на остальных стеках столько не нужно);
4. конвенции документа — эталонные 1:1: паспорт в шапке, раздел 0 не нужен
   (он уже в эталоне), обоснование каждого блока курсивом, `[≈ согласовать]`,
   приложения А (конкуренты + отклонённые ходы), Б (покрытие), В (чек-лист
   внедрения).

Результат: `seo-pages/tehnologii/<slug>.md`. Страницу этот шаг не создаёт и
не меняет.

### 4. SEO-оптимизация контент-документа

Методика [seo-page-optimization](../seo-page-optimization/SKILL.md)
применяется к **файлу** шага 3; «было → станет» вырождается в проверку
документа против ядра (страницы ещё нет):

1. карта «подкластер/интент → блок документа»: каждый P0/P1 покрыт блоком или
   явным объяснением;
2. Title/H1/Description по правилам зон (P0 ближе к началу, цельные
   предложения); уникальность Title/Description в пределах сайта — грепом по
   `*/index.html`; учти решение B0 о бренде в Title;
3. карьерные и исключённые запросы не таргетируются; «передать другим URL» —
   только перелинковка;
4. анти-переспам-проход по всему копирайту;
5. итоговую карту покрытия сохрани — она пойдёт в отчёт шага 8.

### 5. Сверка инфраструктуры

B0 выполнен — стройку пропусти, но сверь, что контракты источника не
дрейфанули (преамбула `docs/texts-tehnologii.md`: поля секции, спец-маркеры)
и что `TECH_META`/`AI_DIRECTIONS_COPY` на месте. Дизайн-система `.tech-*`
утверждена — новые типы блоков вводи только по явному решению пользователя.

### 6. Наполнение и генерация

По чек-листу «Онбординг нового стека» (раздел инфраструктуры выше): секция MD
из `seo-pages/tehnologii/<slug>.md` сверху вниз по каркасу (**кроме позиций
6 и 11**), запись `TECH_META`, копия «Отраслей», hero-картинка, `urls.json`
+1, генерация; `git diff` — п. 8 шпаргалки. Хаб включит новую страницу сам;
сними предохранители взаимных ссылок на опубликованных лендингах (блок 3).

### 7. Верификация

1. Программные проверки — п. 6 шпаргалки (включая греп запрещённых маркеров);
   оформление нейтральных блоков сверь с
   [inner-page-blocks](../inner-page-blocks/SKILL.md) (шапки секций, FAQ);
   ни один блок не «расклеился» в прозу.
2. Локальный сервер (`python -m http.server 8000`): HTTP 200 лендинга **и**
   хаба, рендер без ошибок, читаемость на ширине 320 px (особенно матрица и
   таблица сравнения — мобильная раскладка новая).
3. Панель «Технологии» в меню и футер — на страницах разных разделов и на
   главной (regex-патчи главной могли молча не сработать).

### 8. Перелинковка и отчёт

1. Входящие 2–3 ссылки с релевантных страниц `/uslugi/` (правки
   `docs/texts-usligi.md` + регенерация): для laravel по эталону —
   razrabotka-veb-prilozhenij, modernizaciya-legacy, it-autstaffing; для
   других стеков — по таблице передачи ядра.
2. Отчёт `seo-optimization/tehnologii/<slug>.md` по шаблону
   seo-page-optimization: карта покрытия, каркас с обоснованием отклонений,
   таблица «блок → источник в генераторе», блок «⚠️ Не подтверждены бизнесом —
   подтвердить до деплоя» (все `[≈ согласовать]`), блокеры, план контроля.
3. Индексация (переобход нового URL в Вебмастере, IndexNow) — **только после
