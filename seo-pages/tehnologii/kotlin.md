## Приложение Б. Каннибализация и отклонения от каркаса

Отклонений от эталонного каркаса нет: блоки и их порядок 1:1 с Laravel (интро-панель → матрица «Что строим» → {verdict-cards}+таблица → {eco-map} → {price-models}+таблица → {rescue-cards} → {staff-panel} → {steps} → FAQ×10). Локальные решения, продиктованные стеком, а не отклонения: (1) альтернатива в сравнительной таблице — КРОСС-ПЛАТФОРМА (Flutter/React Native), а не Java, т.к. по выдаче реальная развилка покупателя мобильного приложения — «нативка vs кросс» (code9, o2k явно её обыгрывают); Java закрыта отдельным вопросом FAQ №3 (миграционный интент), кросс-платформа — вопросом №4 и таблицей — обе обязательные формулировки на месте. (2) В матрицу добавлены бэкенд-строка (Ktor/Spring) и Android TV — расширяют Kotlin за пределы «только Android-мобайл», занимая пустую нишу выдачи. РИСК КАННИБАЛИЗАЦИИ с существующими страницами услуг: /uslugi/android-razrabotka/, /uslugi/mobilnaya-razrabotka/, /uslugi/krossplatformennaya-razrabotka/, /uslugi/razrabotka-mvp/ — стековый лендинг таргетирует запрос ТЕХНОЛОГИИ («разработка на Kotlin», «Kotlin на заказ»), а страницы услуг — запрос УСЛУГИ («разработка Android-приложений», «мобильная разработка», «кроссплатформенная разработка»); лендинг не конкурирует за «android приложение заказать», а распределяет посетителя по этим услугам через матрицу (входящая перелинковка со стека на услуги — системообразующая). Предохранитель: упоминания будущих /tehnologii/* (Swift, Flutter, React Native) в тексте даны БЕЗ ссылок — этих URL ещё нет.
